# 新仕様（開発アプリ仕様書）

日報入力デスクトップアプリ（アプリ名「日報入力」／案A: WebView 組み込み / Electron）の仕様・実装をまとめる。
対象システムの既存仕様は [現行仕様.md](現行仕様.md)、プロジェクト背景・要望は [README.md](README.md) を参照。

---

## 1. 目的・方針

- 社内 LAN / VPN 接続端末で、**exe を実行して起動するデスクトップアプリ**から、日報登録画面（`https://rkanri.genech.co.jp/kanri/nippo`）の各 API を実行できるようにする。
- 既存 Web 画面を直接操作せず、アプリ独自の GUI から登録・更新・削除・一覧取得・合計取得・マスタ検索を行う。

### 採用アーキテクチャ（案A: WebView 組み込み）

対象 API は `credentials: include`（Cookie セッション）と**セッション毎に変わる XSRF トークン**を要求するため、外部プロセスからの直接呼び出しは認証の取り回しが難しい。
そこで**アプリ内に対象ページを読み込む WebView を持ち、その（ログイン済み）オリジンのコンテキストで fetch を実行する**方式を採る。これにより Cookie / XSRF が自動的に有効な状態で API を呼べる。

```
┌───────────────────────────── Electron BrowserWindow ─────────────────────────────┐
│  自前UI（src/index.html + renderer.js）        <webview src="…/kanri/nippo">       │
│  ┌───────────────────────────┐                ┌──────────────────────────────┐   │
│  │ 操作パネル（入力・ボタン） │  executeJavaScript(fetch…)                     │   │
│  │ 結果テーブル / JSON        │ ─────────────▶ │ rkanri のページ（ログイン済） │   │
│  │                            │ ◀───────────── │ same-origin fetch + Cookie +  │   │
│  └───────────────────────────┘   結果(JSON)    │ XSRFトークン                  │   │
│                                                 └──────────────┬───────────────┘   │
└────────────────────────────────────────────────────────────────┼──────────────────┘
                                                                  ▼
                                             https://rkanri.genech.co.jp/kanri/nippo/*
```

### 3.1 プロセス構成とセキュリティ設定（main.js）

- `BrowserWindow` の `webPreferences`:
  - `contextIsolation: true` / `nodeIntegration: false`（レンダラーに Node を露出しない）
  - `webviewTag: true`（`<webview>` 埋め込みのため有効化）
  - `preload: preload.js`
- メニューバー非表示（`setMenuBarVisibility(false)`）。
- `setWindowOpenHandler` で webview 内の `window.open` は既定ブラウザへ（アプリ内遷移を防止）。
- レンダラー（renderer.js）は **Node API を使わず**、DOM と `<webview>` 要素の操作のみで完結（安全側）。
- ウィンドウタイトル／アプリ名は **「日報入力」**（`main.js` の `title`、`package.json` の `productName`）。
- `preload.js` は `contextBridge` で最小限の API を公開: `appInfo`（名称・バージョン・対象URL）と `appApi.clearSession()`（再認証用の IPC。§4.4）。
- `main.js` は IPC `clear-session` を受け、`session.fromPartition('persist:rkanri').clearStorageData()` を実行する。
- 画面レイアウトは `body` を flex 縦並び＋`overflow:hidden` とし、上部バーは固定・本文（`web-host`／`main-view`）は `flex:1; min-height:0` で残り高さを埋める。**ページ自体は縦スクロールさせず**、一覧などは各エリア内でスクロールする（高さの `calc(100% - 44px)` 固定は廃止）。

## 4. 中核処理：ページコンテキストでの API 実行

`renderer.js` の `callApi(endpoint, body, label)` が中核。手順は以下。

1. `<webview>`（id=`rk`）が `dom-ready` 済みか確認（未ログイン/未読込なら中断）。
2. `buildInjection(endpoint, body)` で、対象ページ内で実行する fetch コードを生成。
3. `rk.executeJavaScript(code)` で**対象ページのオリジン上**で実行。戻り値（Promise）を await。
4. 結果を結果テーブル / JSON 欄に表示。

### 4.1 注入する fetch（対象ページ内で実行される）

- URL: 相対パス `/kanri/nippo/<endpoint>`（same-origin）
- `method: POST` / `credentials: 'include'`
- ヘッダー:
  - `accept: application/json, text/plain, */*`
  - `content-type: application/json`
  - `x-xsrf-token`: **`document.cookie` の `XSRF-TOKEN` を URL デコードした値**
    - Laravel の `XSRF-TOKEN` Cookie は非 httpOnly のため JS から読める。
    - セッション Cookie は `credentials: include` により自動送出。
- `body`: 送信オブジェクトの JSON 文字列
- 戻り値: `{ ok, status, url, hasToken, data }`（`data` は JSON パース、失敗時は生テキスト）

### 4.2 ログイン切れの検出

- レスポンスが文字列で `<html` / `<!doctype` / `login` を含む場合、未ログインと判定し、認証画面（`showAuth`）に戻して「認証画面でログインしてください」と案内（接続ステータスも警告表示）。

#### 実行結果の表示方針（`callApi`）

- **成功時はステータス欄（`outStatus`）に何も表示しない**（「OK (200)」等は出さない）。成功トーストのみ（自動処理は `silent` で抑制）。
- **エラー時はサーバー応答をそのまま「ポップアップ（toast）」で表示**する（canned な日本語には変換しない）。ステータス欄にも同内容を出す。HTTP エラー・`{status}` が `success` 以外・実行例外・ログイン切れのいずれもトーストで通知する。
- 生レスポンス JSON は `<details>` に常時保持。

### 4.3 接続ステータス表示

- webview の `did-start-loading` / `did-stop-loading` / `dom-ready` / `did-fail-load` を監視し、上部のドット（緑=OK / 橙=読込中 / 赤=失敗）とテキストで表示。
- **緑化は `did-stop-loading`（読み込み完了）で判定**する。Microsoft SSO は複数回リダイレクトし、その都度 `did-start-loading`（橙）が再発火するため、`dom-ready` だけでは橙のままに見える。読み込みが本当に止まった時点で緑にする。
- `did-fail-load` の `errorCode === -3`（ABORTED、リダイレクト等）は無視。

### 4.4 再認証（セッション消去）

- 上部バーの「再認証」（`reauth`）で、webview パーティション `persist:rkanri` のセッション（Cookie 等）を消去し、認証画面に戻して webview を再読込する。
- Cookie が消えるため rkanri は未ログインとなり、**Microsoft SSO のログイン画面から入り直せる**（別アカウントへの切替も可能）。
- 消去はメインプロセスで実行する（`preload` の `appApi.clearSession()` → IPC `clear-session` → `session.fromPartition('persist:rkanri').clearStorageData()`）。
- 再認証時は本人自動セットのフラグ（`tantoAutoDone`）もリセットし、次回ログインで担当者を再判定する。

## 5. 画面仕様（UI）

**モーダル（ポップアップ）ベースのフロー**。メイン画面は日報一覧を中心にし、登録・マスタ検索はモーダルで行う（[README.md](README.md)「希望する画面フロー」準拠）。

- webview は **常時 DOM 上に保持**し、アプリ画面表示中はオフスクリーンへ退避（`body.app .web-host` を `position:fixed; left:-10000px` でサイズ維持）することで、`executeJavaScript` 実行に必要なオリジン・セッションを保持する。

### 5.0.1 上部バー（全画面共通）

- **ページ名タイトル**（`pageTitle`）: **「ログイン」か「日報一覧」の2パターンのみ**。
  - 認証画面 → **ログイン**／アプリ画面 → **日報一覧**。
  - 登録モーダル・ZOOM（マスタ一覧）モーダルを開いてもヘッダーは変更しない（日報一覧のまま）。
  - `updateHeaderTitle()` は `body` の状態（auth/app）だけで判定する。
- 接続ステータス（ドット + テキスト）。
- 「↻ 再読込」ボタン（webview リロード）。アプリ画面での再読込時は、日報一覧の担当者を**ログイン担当者に戻し**（`restoreLoginTanto`。§5.2）、読み込み完了（`did-stop-loading`）後にその担当者で一覧を取り直す（`pendingListReload`）。
- 「**再認証**」ボタン（再読込の右）: webview セッションを消去して Microsoft SSO からログインし直す（§4.4）。

### 5.1 ① 認証画面（`body.auth`）

- ログイン導線ヒント + `<webview>`（`src=https://rkanri.genech.co.jp/kanri/nippo`, `partition="persist:rkanri"`）。
- 「**ログインを確認してアプリを開く**」ボタン（`tryEnter`）。
- **自動遷移**: webview の `did-stop-loading`/`dom-ready` 時、URL が対象ページ（`/kanri/nippo`）であれば `tryEnter(silent)` を自動実行。`getNippoList` が配列を返せば認証成功とみなしメイン画面へ遷移。HTML/login が返れば認証画面に留まる。
- **遷移マスク**（`enterMask`）: ログイン完了後、対象ページの描画（＝既存システム画面）が一瞬見えるのを防ぐため、`did-navigate` で URL が対象ページに変わった瞬間（描画前）に全画面のローディングマスクで覆う。API 取得〜メイン画面表示まで覆い続け、`enterApp`／`showAuth` で解除する。ログイン画面（対象外 URL）ではマスクを外してログイン UI を表示する。

### 5.2 ② メイン画面（`body.app`）

- **ツールバー**: 担当者コード(`tanto`, **readonly**) / 担当者名(`tantoName`, readonly) / **「検索」**（担当者マスタ検索 ⑥ を開く。`btnTantoSearchMain`） / 「↻ 一覧更新」(`getNippoList`) / **「＋ 新規登録」** / **「一括コピー」** / **「一括削除」**。（「再認証」は上部バーへ移動＝§5.0.1）
  - 担当者コードは手入力不可（readonly）。担当者は**担当者名の右の「検索」ボタン**（⑥ 担当者マスタ一覧）から選択して設定する。
- **日報一覧テーブル**（`listTable`）: `getNippoList` の結果を表示（§5.6 の表示列）。行はクリック可能。**並び順は「日付 降順 → 作業時間 降順 → 案件コード 降順」**（`sortNippoList`）でクライアント側ソートして表示する。
  - **選択列**: 日付列の左に**チェックボックス列**を持つ（ヘッダは全選択）。選択は `selectedNippo`（日報コードの Set）で管理し、一覧再取得時にクリアする。チェック操作では行クリック（編集）を発火させない（`stopPropagation`）。
  - **Ctrl（Mac は Cmd）+ 行クリック**: 編集モーダルを開かず、その行の選択をトグルする（チェックボックス以外の場所でも選択可能）。通常クリックは従来どおり編集モーダルを開く。

#### 一括コピー（`openBulkCopy` / `doBulkCopy`）

- 一覧で選択した日報を、指定日に**新規登録（コピー）**する。ボタン押下で一括コピーモーダル（`bulkCopyModal`）を開く。
- モーダル: 日付種別（**単一 / 範囲**ラジオ）＋日付（`bulkDateFrom`／範囲時は `bulkDateTo`。カレンダー入力のみ）＋「コピー実行」。選択件数を表示。
- **単一**: 指定日に、選択した各日報を新規登録。
- **範囲**: `bulkDateFrom`〜`bulkDateTo`（両端含む）を1日ずつ列挙（`dateRange`、最大366日）し、**選択した各日報 × 各日**を新規登録。開始日が終了日より未来になった場合は**終了日＝開始日**に自動補正する。
- 送信は `registData`（`nippoCd` 空＝新規）。担当者・案件・作業内容・時間・進捗・遅れ報告・その他報告は元の日報の値をコピーし、日付のみ対象日に置換。
- 実行後に成功/失敗件数をトースト表示し、一覧を再取得。

#### 一括削除（`doBulkDelete`）

- チェックした日報を確認ダイアログの後、`deleteData` で順次一括削除し、成功/失敗件数をトースト表示して一覧を再取得する。
- 下部にステータス行（`outStatus`）と生レスポンス JSON（`<details>`）。
- **担当者の扱い**: メイン（日報一覧）の担当者（`tanto`/`tantoName`）は、初期はログイン担当者（本人自動セット）。**ツールバーの「検索」ボタン（⑥ 担当者マスタ一覧）から選択して切り替えられる**（選択後に一覧を取り直す）。担当者コードは readonly で手入力不可。登録モーダルの担当者マスタ検索（⑥）で担当者を変えても**メインは変更しない**（登録対象＝`regTanto` のみ変更。§5.3）。担当者モーダルは開いた場所によって設定先が変わる（`tantoPickTarget`＝`main`/`reg`。§5.5）。**「↻ 再読込」でログイン担当者に戻す**: 本人自動セットの結果を `loginTanto` に保持し、再読込時に `restoreLoginTanto` で日報一覧の担当者を `loginTanto` へ戻す（自動セット前は HTML 初期値）。

#### 本人の自動セット（`autoSetTantoOnce`）

- 認証成功後（アプリ遷移時）、**初回のみ**ログイン中ユーザーの担当者を自動判定してツールバーに設定する。
- 手順:
  1. webview（rkanri ページ）の `<input>` を走査し、値が担当者コード形式（`/^P-\d{6}$/`）の最初の1件を取得（`AUTO_TANTO_PROBE`）。対象ページは担当者コードを input に保持しているため、その値＝ログイン中ユーザーの担当者コードとみなす。
  2. 取得コードを `tanto` に設定し、`getTantoList {svalue: コード}`（トースト抑制）で `名称1` を解決して `tantoName` に設定。
  3. 既定コードから変わった場合は一覧を取り直す。
- **フォールバック**: 取得・解決に失敗した場合は既定値（HTML の初期値 `P-000184`）のまま。各自が担当検索（⑥）で選択すればよい。初期値はログイン判定の初回 `getNippoList` プローブにも使う。
- 一度自動セットしたら（`tantoAutoDone`）、以降の再認証では上書きしない（ユーザーの手動選択を尊重）。
- ⚠ 前提: 「rkanri の担当者コード input にはログイン中ユーザー自身のコードが入る」ことを前提とする。サーバーが全ユーザー共通の既定値を返す実装だと本人にならないため、**別ユーザーでの実機確認を推奨**。うまく取れない場合は取得元（DOM セレクタ／ページ内グローバル変数等）を実ページに合わせて調整する。

### 5.3 ③④ 登録モーダル（`registModal`／登録エリアのみ）

- **③ 修正モード**: 一覧の行クリック（`openEditModal`）で開く。行データを各項目へ反映し `nippoCd` あり＝編集モード（バッジ「編集モード」）。
- **④ 新規モード**: 「＋ 新規登録」（`openNewModal`）で開く。フォーム初期化・日付＝当日・`nippoCd` 空＝新規モード（バッジ「新規モード」）。
- 構成項目:
  - ヘッダ: タイトル「日報登録」 + モードバッジ / 日報コード表示（`nippoCdView`、新規時は「新規登録」） / **合計値（`totalVal`、日報コードの右）** / 閉じる(✕)
  - 日付(`date`, **`<input type="date">` のカレンダー入力のみ／キーボード直接入力は不可**) + 一昨昨日・一昨日・昨日・今日
  - 幅の統一: 日付・担当者コード(`regTanto`)・案件コード(`ankenCd`)・作業内容(`sagyoNaiyo`) は共通クラス `.w-field`（150px）で幅を揃える。
  - **担当者**: `regTanto`(readonly) / `regTantoName`(readonly) / **[検索]**（⑥ 担当者モーダルを開く）。**この担当者が登録対象**（`registData` の `tanto`／`getTotal`・案件履歴の対象）。新規時の既定はログイン担当者（`syncRegTanto`）、修正時は行の担当者。
  - **案件コード**(`ankenCd`, readonly) / **[検索]**（⑤ 案件モーダルを開く）
  - 取引先名(`torihikisakiMei`, readonly) / 案件名(`ankenMei`, readonly)
  - 作業内容(`sagyoNaiyo`, select) / 作業時間(`sagyozikan`) / 作業進捗(`sagyoShinchoku`, select)
  - 遅れ報告(`okureHokoku`) / その他報告(`sonotaHokoku`)
  - フッタ: 登録/更新(`registData`) / 削除(`deleteData`) / コピー（緑ボタン、**編集モードのみ表示**） / **リセット**（閉じるはヘッダーの ✕ のみ）
  - **リセット**（`resetForm`）: モーダルを**開いた時点の状態に戻す**（＝編集のリセット）。新規モードならデフォルト表示、編集モードなら選択された未編集の値。開いた時に `takeSnapshot()` で全項目（`SNAPSHOT_FIELDS`＝送信11項目＋`tanto`/`tantoName`）を記録し、リセットで復元して合計も取り直す。
- 日付ボタンは現在日付 +offset を設定する。**画面内部は `yyyy-mm-dd`（`type="date"` の形式）で保持し、API 送信時に `yyyy/m/d`（ゼロ埋めなし）へ変換**する（`toApiDate`）。編集モードで一覧の `yyyy/m/d` を読み込む際は `toInputDate` で `yyyy-mm-dd` へ変換して入力欄へ設定する。
- **合計は自動取得**: 登録モーダルを開いた時（新規＝当日／修正＝その日報の日付）に `getTotal` を自動実行して `totalVal` に表示する（`doTotal`、トースト抑制）。日付ボタン押下・日付手入力（`change`）でも再取得する。**「合計取得」ボタンは廃止**（合計値ラベルのみ表示）。
- `tanto`/`tantoName` の正はメイン画面側。登録モーダルの `regTanto`/`regTantoName` は表示ミラーで、担当検索（⑥）で選択するとメイン側と両方更新（`syncRegTanto`）。送信 `tanto` はメイン側の値を使う。

### 5.4 ⑤ 案件マスタ検索モーダル（`ankenModal`）

- 検索テキスト(`svalue`) + 「検索」(`gethistory {tanto, svalue}`) / 「履歴」(`gethistory {tanto}`)。
- **開いた時に自動で「履歴」(`gethistory {tanto}`)を実行して初期表示**する（`svalue` はクリア）。「検索」ボタンは再検索用。
- 結果テーブル（`ankenTable`）を表示。行クリック(`pickAnken`)で **`ankenCd`/`torihikisakiMei`/`ankenMei`** を登録モーダルへ設定し、案件モーダルを閉じる。

### 5.5 ⑥ 担当者マスタ検索モーダル（`tantoModal`）

- 検索テキスト(`svalue2`) + 「検索」(`getTantoList {svalue}`)。
- **開いた時に自動で全件(`getTantoList {svalue:""}`)を実行して初期表示**する（`svalue2` はクリア）。「検索」ボタンは再検索用。
- **設定先の切り替え（`openTantoModal(target)` / `tantoPickTarget`）**: 同じモーダルを2箇所から開き、行クリック(`pickTanto`)時の設定先を切り替える。
  - `target='main'`（メインツールバーの「検索」）: **メイン（日報一覧）の `tanto`/`tantoName`** を選択行（`key`/`名称1`）に設定し、モーダルを閉じて**一覧を取り直す**（`loadList`）。
  - `target='reg'`（登録モーダルの「検索」）: **登録モーダルの担当者のみ**（`regTanto`/`regTantoName`）を変更しモーダルを閉じる。**メインの `tanto`/`tantoName` は変更しない**。変更後は登録モーダルの合計（`doTotal`）を取り直す。

### 5.6 テーブル表示の共通仕様

`renderTableInto(container, endpoint, data, onRowClick)` で任意コンテナへ描画する（メイン一覧・各モーダル一覧で共用）。

- **表示列の絞り込み**: エンドポイント別ホワイトリスト（`DISPLAY_COLS`）で該当列のみ表示。`getNippoList` は実測25フィールド返るため（[現行仕様.md](現行仕様.md) §7.4）、日付 / 案件コード / 取引先名 / 案件名 / 作業時間 / 作業内容 / 作業進捗 / その他報告事項 に絞る。未定義／非該当は全列にフォールバック。
- **見出しラベル**: `HEADER_LABELS` で日本語化。`getTantoList` の `key`→`担当者コード` / `名称1`→`担当者名`（同 §6.2）。
- **非配列応答は既存テーブルを維持**: `getTotal`（スカラー）・`registData`/`deleteData`（`{status}`）は再描画せず直前の表示を残す。空配列 `[]` のみ「（データなし）」表示。
- **行データは全フィールド保持**。表示列を絞っても行クリック時の反映（`日報コード`・`遅れ報告` 等の非表示列を含む）に影響しない。

### 5.7 選択肢

**空欄の選択肢は設けず、必ずいずれかが選択される**（現行仕様の先頭空欄は廃止）。

- 作業内容(`sagyoNaiyo`): `1.管理` / `2.PJ会議` / `2.PJレビュー` / `3.調査・分析` / `4.要件定義` / `5..設計` / `6.開発・構築` / `7.テスト` / `8.ドキュメント作成` / `9.リリース対応` / `※作業待ち` / `※プレ・見積` / `※不具合` / `※問い合わせ` / `※社内会議` / `※レビュー支援` / `※学習・教育` / `※その他`（既定＝先頭 `1.管理`）
- 作業進捗(`sagyoShinchoku`): `順調` / `やや遅れ` / `遅れ`（既定 `順調`）
- クリア時も空欄には戻さず、作業内容＝先頭 / 作業進捗＝`順調` にリセットする。

### 6.1 登録・更新の送信項目（全11）

`nippoCd, tanto, date, ankenCd, torihikisakiMei, ankenMei, sagyoNaiyo, sagyozikan, sagyoShinchoku, okureHokoku, sonotaHokoku`

### 6.2 業務ロジック（現行準拠）

- **モード判定**: `nippoCd` が空=新規モード、値あり=編集モード（バッジ表示）。
- **コピー**: `nippoCd` を空にするのみ（他項目は保持）→ 新規化。
- **登録/更新成功後**: フォーム初期化 → `getNippoList` で一覧再取得。
- **削除**: `nippoCd` 空なら中断。確認ダイアログ（「削除します。元に戻せませんのでご注意を」）承認後に実行 → 初期化 → 一覧再取得。

#### 成否判定（`status` ベース）

`registData` / `deleteData` は HTTP 200 でもボディ `{"status":"success"}` を返す（[現行仕様.md](現行仕様.md) §8.3/§8.4）。本アプリは現行画面でコメントアウトされていた `status` 判定を**実装として有効化**する。

- `isBizOk(r)` = HTTP OK **かつ**（応答が `{status:...}` 形式なら `status === "success"`）。
- `isBizOk` が真のときのみ「成功」toast → フォーム初期化 → 一覧再取得を実行。
- HTTP 200 でも `status !== "success"` の場合は**失敗扱い**（フォーム保持・一覧再取得なし）。
- **エラー時はサーバー応答をそのまま表示する**（canned な日本語メッセージに変換しない）。エラー時の応答形（`status` 値や構造）が未確認のため、`rawText()` で応答本体（オブジェクトは JSON 文字列化／文字列はそのまま）をステータス欄と toast に出力する。生レスポンスは従来どおり `<details>` にも表示。
- 一覧系（`getNippoList`/`gethistory`/`getTantoList`＝配列）・`getTotal`（スカラー）は `{status}` を持たないため、従来どおり HTTP OK で成功扱い。HTTP エラー（非 2xx）時も応答本体をそのまま表示。

### 6.3 入力バリデーション（`checkError` 相当）

上から評価し、最初に該当したものを toast 表示して中断。

1. `regTanto` 空 → 担当者が未入力です
2. `ankenCd` 空 → 案件が未入力です
3. `date` 空 → 日付が未入力です
4. `date` 不正（`new Date` が NaN） → 日付が不正です
5. `sagyoNaiyo` 空 → 作業内容が未入力です
6. `sagyozikan` 空 → 作業時間が未入力です
7. `sagyozikan` 非数値 → 作業時間が不正です
8. `sagyoShinchoku` 空 → 作業進捗が未入力です
9. `sonotaHokoku` 空 → その他報告事項が未入力です
10. `sagyoShinchoku` が「順調」以外かつ `okureHokoku` 空 → 作業進捗が順調以外の場合は遅れ報告を入力してください

### 7.1 単一 exe（electron-builder）に関する既知の制約

- `npm run dist` / `dist:nsis` は署名ツール（winCodeSign）展開時に、アーカイブ内の macOS 用 `.dylib` シンボリックリンクを作ろうとし、**Windows のシンボリックリンク作成権限**（管理者 or 開発者モード）が無いと `7za` が「必要な特権を保有していません」で失敗することがある（この `.dylib` は Windows ビルドには不要）。

- 上記キャッシュを削除すると再発する。その場合は手動展開をやり直すか、開発者モード ON／管理者権限ターミナルでビルドする。

## 8. セキュリティ・運用上の留意点

- **認証は本人のブラウザセッションに依存**。アプリは本人の権限・データ範囲でのみ動作する。認証情報をアプリが保存・送信することはない（Cookie は webview の `persist:rkanri` パーティションが保持）。
- **XSRF トークンはハードコードしない**。実行時に Cookie から都度取得する。
- 社内ネットワーク（LAN/VPN）接続時のみ対象ページに到達できる前提。オフライン/社外では webview 読込が失敗し、接続ステータスが赤表示になる。
- `registData` はサーバー応答の `status` によるエラー判定が現行画面ではコメントアウトされている（[現行仕様.md](現行仕様.md) §8.3）。本アプリでは HTTP ステータスと本文で成否を判定しているが、業務エラー（重複・制限超過等）の詳細レスポンス仕様は要確認。
