# 新仕様（開発アプリ仕様書）

日報入力デスクトップアプリ（アプリ名「日報入力」／案A: WebView 組み込み / Electron）の仕様・実装をまとめる。
対象システムの既存仕様は [現行仕様.md](現行仕様.md)、プロジェクト背景・要望は [README.md](README.md) を参照。

---

## 1. 目的・方針

- 社内 LAN / VPN 接続端末で、**exe を実行して起動するデスクトップアプリ**から、日報登録画面（`https://rkanri.genech.co.jp/kanri/nippo`）の各 API を実行できるようにする。
- 既存 Web 画面を直接操作せず、アプリ独自の GUI から登録・更新・削除・一覧取得・合計取得・マスタ検索を行う。

### 採用アーキテクチャ（案A: WebView 組み込み）

対象 API は `credentials: include`（Cookie セッション）と**セッション毎に変わる XSRF トークン**を要求するため、外部プロセスからの直接呼び出しは認証の取り回しが難しい。
そこで**アプリ内に対象ページを読み込む WebView を持ち、その（ログイン済み）オリジンのコンテキストで fetch を実行する**方式を採る。これにより Cookie / XSRF が自動的に有効な状態で API を呼べる。

```
┌───────────────────────────── Electron BrowserWindow ─────────────────────────────┐
│  自前UI（src/index.html + renderer.js）        <webview src="…/kanri/nippo">       │
│  ┌───────────────────────────┐                ┌──────────────────────────────┐   │
│  │ 操作パネル（入力・ボタン） │  executeJavaScript(fetch…)                     │   │
│  │ 結果テーブル / JSON        │ ─────────────▶ │ rkanri のページ（ログイン済） │   │
│  │                            │ ◀───────────── │ same-origin fetch + Cookie +  │   │
│  └───────────────────────────┘   結果(JSON)    │ XSRFトークン                  │   │
│                                                 └──────────────┬───────────────┘   │
└────────────────────────────────────────────────────────────────┼──────────────────┘
                                                                  ▼
                                             https://rkanri.genech.co.jp/kanri/nippo/*
```

### 3.1 プロセス構成とセキュリティ設定（main.js）

- `BrowserWindow` の `webPreferences`:
  - `contextIsolation: true` / `nodeIntegration: false`（レンダラーに Node を露出しない）
  - `webviewTag: true`（`<webview>` 埋め込みのため有効化）
  - `preload: preload.js`
- メニューバー非表示（`setMenuBarVisibility(false)`）。
- `setWindowOpenHandler` で webview 内の `window.open` は既定ブラウザへ（アプリ内遷移を防止）。
- レンダラー（renderer.js）は **Node API を使わず**、DOM と `<webview>` 要素の操作のみで完結（安全側）。
- ウィンドウタイトル／アプリ名は **「日報入力」**（`main.js` の `title`、`package.json` の `productName`）。
- `preload.js` は `contextBridge` で `appInfo`（名称・バージョン・対象URL）と `appApi` を公開する。`appApi` は **再認証**（`clearSession()`。§4.4）、**本人特定**（`getCapturedEmail()` / `getIdentity()` / `setIdentity()` / `clearIdentity()`。§5.2）、**自動更新**（`checkUpdate()` / `startUpdate()` / `quitAndInstall()` / `onUpdateStatus(cb)`。§7.2）の各 IPC を提供する。
- `main.js` は IPC `clear-session` を受け、`session.fromPartition('persist:rkanri').clearStorageData()` に加えて捕捉メール・`identity.json` を破棄する（§4.4）。
- 画面レイアウトは `body` を flex 縦並び＋`overflow:hidden` とし、上部バーは固定・本文（`web-host`／`main-view`）は `flex:1; min-height:0` で残り高さを埋める。**ページ自体は縦スクロールさせず**、一覧などは各エリア内でスクロールする（高さの `calc(100% - 44px)` 固定は廃止）。

## 4. 中核処理：ページコンテキストでの API 実行

`renderer.js` の `callApi(endpoint, body, label)` が中核。手順は以下。

1. `<webview>`（id=`rk`）が `dom-ready` 済みか確認（未ログイン/未読込なら中断）。
2. `buildInjection(endpoint, body)` で、対象ページ内で実行する fetch コードを生成。
3. `rk.executeJavaScript(code)` で**対象ページのオリジン上**で実行。戻り値（Promise）を await。
4. 結果を結果テーブル / JSON 欄に表示。

### 4.1 注入する fetch（対象ページ内で実行される）

- URL: 相対パス `/kanri/nippo/<endpoint>`（same-origin）
- `method: POST` / `credentials: 'include'`
- ヘッダー:
  - `accept: application/json, text/plain, */*`
  - `content-type: application/json`
  - `x-xsrf-token`: **`document.cookie` の `XSRF-TOKEN` を URL デコードした値**
    - Laravel の `XSRF-TOKEN` Cookie は非 httpOnly のため JS から読める。
    - セッション Cookie は `credentials: include` により自動送出。
- `body`: 送信オブジェクトの JSON 文字列
- 戻り値: `{ ok, status, url, hasToken, data }`（`data` は JSON パース、失敗時は生テキスト）

### 4.2 ログイン切れの検出

- レスポンスが文字列で `<html` / `<!doctype` / `login` を含む場合、未ログインと判定し、認証画面（`showAuth`）に戻して「認証画面でログインしてください」と案内（接続ステータスも警告表示）。

#### 実行結果の表示方針（`callApi`）

- **成功時はステータス欄（`outStatus`）に何も表示しない**（「OK (200)」等は出さない）。成功トーストのみ（自動処理は `silent` で抑制）。
- **エラー時はサーバー応答をそのまま「ポップアップ（toast）」で表示**する（canned な日本語には変換しない）。ステータス欄にも同内容を出す。HTTP エラー・`{status}` が `success` 以外・実行例外・ログイン切れのいずれもトーストで通知する。
- 生レスポンス JSON は `<details>` に常時保持。

### 4.3 接続ステータス表示

- webview の `did-start-loading` / `did-stop-loading` / `dom-ready` / `did-fail-load` を監視し、上部のドット（緑=OK / 橙=読込中 / 赤=失敗）とテキストで表示。
- **緑化は `did-stop-loading`（読み込み完了）で判定**する。Microsoft SSO は複数回リダイレクトし、その都度 `did-start-loading`（橙）が再発火するため、`dom-ready` だけでは橙のままに見える。読み込みが本当に止まった時点で緑にする。
- `did-fail-load` の `errorCode === -3`（ABORTED、リダイレクト等）は無視。

### 4.4 再認証（セッション消去）

- 上部バーの「再認証」（`reauth`）で、webview パーティション `persist:rkanri` のセッション（Cookie 等）を消去し、認証画面に戻して webview を再読込する。
- Cookie が消えるため rkanri は未ログインとなり、**Microsoft SSO のログイン画面から入り直せる**（別アカウントへの切替も可能）。
- 消去はメインプロセスで実行する（`preload` の `appApi.clearSession()` → IPC `clear-session`）。`clear-session` は `session.fromPartition('persist:rkanri').clearStorageData()` に加え、**本人特定の捕捉メール（`capturedEmail`）と永続化 identity（`userData/identity.json`）も破棄**する（別アカウントへの切替に対応。§5.2）。
- 再認証時はレンダラー側でも `loginTanto` を破棄し、認証方式（`authMethod`）を `sso` に戻す（手動入力モードの残りを解除）。次回ログインで本人を取り直す（§5.2）。
- **ブラウザ表示版**（`IS_BROWSER`）の再認証は `POST /api/logout` でローカルサーバー側のセッション・identity を破棄し、認証画面（「ログイン」ボタン）へ戻す（§11.4）。

### 5.1 ① 認証画面（`body.auth`）

- ログイン導線ヒント + `<webview>`（`src=https://rkanri.genech.co.jp/kanri/nippo`, `partition="persist:rkanri"`）。
- **フッター（`auth-actions`）**: 中央に「**ログインを確認してアプリを開く**」ボタン（`tryEnter`）、**右端に [更新ボタン `btnUpdateLogin`] [現在のバージョン `verLabelLogin`]**（左右均等スペーサーで開くボタンを中央維持）。ログイン前でも更新確認・適用ができる（更新照会は GitHub Releases への問い合わせで webview のログイン状態に依存しない。§7.2）。
- **自動遷移**: webview の `did-stop-loading`/`dom-ready` 時、URL が対象ページ（`/kanri/nippo`）であれば `tryEnter(silent)` を自動実行。`getNippoList` が配列を返せば認証成功とみなしメイン画面へ遷移。HTML/login が返れば認証画面に留まる。
- **遷移マスク**（`enterMask`）: ログイン完了後、対象ページの描画（＝既存システム画面）が一瞬見えるのを防ぐため、`did-navigate` で URL が対象ページに変わった瞬間（描画前）に全画面のローディングマスクで覆う。API 取得〜メイン画面表示まで覆い続け、`enterApp`／`showAuth` で解除する。ログイン画面（対象外 URL）ではマスクを外してログイン UI を表示する。
- **手動入力パネル**（`manualAuth`。認証方式 `authMethod`＝`manual`）: SSO にはログインできたが、ログインメール・永続化情報のいずれからも本人（`名称4`）を特定できなかった場合に表示するフォールバック UI（`showManualAuth`）。webview をオフスクリーンへ退避し（`html.authmethod-manual`）、白背景の中央パネルにアイコン・案内文（`manualAuthMsg`＝「自動認証できませんでした。ユーザーIDを入力して認証してください。」）・ユーザーID 入力欄（`manualUserId`）・「認証」ボタン（`btnManualAuth`→`submitManualId`）を出す。詳細は §5.2。

### 5.2 ② メイン画面（`body.app`）

- **ツールバー（`main-toolbar`）**: 同一パネル内に**3段**で配置する（担当者関連は上部バーへ移設済み＝§5.0.1／「再認証」も上部バー）。
  - **1段目**: **表示モード切替アイコンボタン**（`btnToggleAll`＝全件⇄月の切替。§5.2.1）/ **モード文言**（`modeLabel`） / **年月シフト**（`monthNav`＝◀ `btnPrevMonth` / 年月カレンダー `filterMonth` / ▶ `btnNextMonth`） / （spacer） / 「↻ 一覧更新」(`btnGetList`＝`getNippoList`) / **「新規登録」** / **「一括コピー」** / **「一括変更」**（`btnBulkEdit`。§5.2.3） / **「一括削除」**。
  - **2段目**: 絞り込み — 案件コード(`fCode`) / 取引先名(`fTorihiki`) / 案件名(`fAnken`)（いずれもテキスト・部分一致）。
  - **3段目**: 絞り込み — 作業内容(`fNaiyo`, select) / 作業進捗(`fShinchoku`, select) / その他報告(`fSonota`, text) / 「クリア」(`btnClearFilter`、**右端**)。
  - 表示モードの詳細は §5.2.1、絞り込みの詳細は §5.2.2。
- **日報一覧テーブル**（`listTable`）: `getNippoList` の結果を表示（§5.6 の表示列）。行はクリック可能。**並び順は「日付 降順 → 作業時間 降順 → 案件コード 降順」**（`sortNippoList`）でクライアント側ソートして表示する。
  - **選択列**: 日付列の左に**チェックボックス列**を持つ（ヘッダは全選択）。選択は `selectedNippo`（日報コードの Set）で管理し、一覧再取得時・絞り込み変更時・表示モード切替・年月シフト時（いずれも `applyListFilter` 経由）にクリアする。チェック操作では行クリック（編集）を発火させない（`stopPropagation`）。
  - **Ctrl（Mac は Cmd）+ 行クリック**: 編集モーダルを開かず、その行の選択をトグルする（チェックボックス以外の場所でも選択可能）。通常クリックは従来どおり編集モーダルを開く。
  - **作業進捗による行の強調**: `作業進捗` が「やや遅れ」の行は薄い赤（`tr.delay-warn`）、「遅れ」の行はやや濃い赤（`tr.delay-late`）で**行全体**を着色する（濃度は 遅れ > やや遅れ）。`renderTableInto` が行の `作業進捗` を見て付与する。

#### 5.2.1 表示モード（月表示 / 全件表示）

一覧を「当月のみ」か「全件」で切り替える。API 再取得はせず、取得済みの全件（`allNippoRows`）に対しクライアント側で絞り込む（`applyListFilter`）。

- **初期状態は月表示モードで、当月**（起動時の年月）で絞り込む（`initListFilter`）。
- **モード切替アイコンボタン**（`btnToggleAll`）: 押下で月表示⇔全件表示をトグル（`setListFilterMode`）。ボタンは**アイコン表示**（切替を表す上下2本矢印のインライン SVG。CSP のため外部画像は使わない）で文言は持たず、現在モードは右隣の文言（`modeLabel`）で示す。ツールチップは切替先（「全件表示へ切替」/「月表示へ切替」）。
  - **月表示モード**: `modeLabel`＝「月表示モード」、年月シフト（`monthNav`）を**表示**。
  - **全件表示モード**: `modeLabel`＝「全件表示モード」、年月シフト（`monthNav`）を**非表示**。
- **年月シフト**（`monthNav`）: ◀（`shiftFilterMonth(-1)`）/ 年月カレンダー（`filterMonth`＝`<input type="month">`）/ ▶（`shiftFilterMonth(1)`）。年月は**カレンダー選択のみ**でキーボード直接入力は不可（Tab を除き `keydown` を抑止）。◀▶ は年跨ぎに対応。
- 月フィルタは `日付` の年月一致（`matchesMonth`）で判定。全件表示モードでは月判定を無効化する。

#### 5.2.2 一覧の絞り込み（検索条件）

ツールバー2・3段目の各条件で一覧をさらに絞り込む（表示モードと**併用**。`applyListFilter` → `matchesFieldFilters`）。入力・選択のたびに即時反映する。

- **部分一致（テキスト）**: 案件コード(`fCode`) / 取引先名(`fTorihiki`) / 案件名(`fAnken`) / その他報告(`fSonota`)。空欄は無条件。
- **完全一致（プルダウン）**: 作業内容(`fNaiyo`) / 作業進捗(`fShinchoku`)。先頭に「**全て**」（空値＝無条件）を置き、選択肢は登録モーダルと同じ定数（§5.7）を用いる。
- **クリア**（`btnClearFilter`、3段目右端）: 全条件を空へ戻して再フィルタ（`clearFieldFilters`）。年月・表示モードは対象外。
- **列の縦揃え**: 案件コード↔作業内容・取引先名↔作業進捗の入力欄が縦に揃うよう、ラベルは固定幅（右寄せ）とし、案件コード入力幅は作業内容プルダウンの実幅に合わせる（`alignFilterColumns`）。案件名↔その他報告の縦揃えは行わない。

#### 一括コピー（`openBulkCopy` / `doBulkCopy`）

- 一覧で選択した日報を、指定日に**新規登録（コピー）**する。ボタン押下で一括コピーモーダル（`bulkCopyModal`）を開く。
- モーダル: 日付種別（**単一 / 範囲**ラジオ）＋日付（`bulkDateFrom`／範囲時は `bulkDateTo`。カレンダー入力のみ）＋「コピー実行」。選択件数を表示。
- **単一**: 指定日に、選択した各日報を新規登録。
- **範囲**: `bulkDateFrom`〜`bulkDateTo`（両端含む）を1日ずつ列挙（`dateRange`。内部安全弁は 400 件で打ち切り）し、**選択した各日報 × 各日**を新規登録。日数の上限（最大366日）は呼び出し側 `doBulkCopy` で判定する。開始日が終了日より未来になった場合は**終了日＝開始日**に自動補正する。
- 送信は `registData`（`nippoCd` 空＝新規）。担当者・案件・作業内容・時間・進捗・遅れ報告・その他報告は元の日報の値をコピーし、日付のみ対象日に置換。
- 実行後に成功/失敗件数をトースト表示し、一覧を再取得。

#### 5.2.3 一括変更（`openBulkEdit`。変更 / 個別変更 / 一括置換）

一覧で選択した日報（`selectedRows`）に対し、**既存日報の一括更新**を行う。ボタン押下で一括変更モーダル（`bulkEditModal`）を開き（選択0件・参照のみモードはトーストで中断）、選択件数（`bulkEditCount`）を表示。**3タブ**（`switchBulkTab`。既定は「一括変更」）で構成し、各タブに対応する実行ボタンをフッターに出し分ける。いずれも更新は `registData`（`nippoCd`＝日報コード）で1件ずつ送信し、成功/失敗（/変更なし）件数をトースト表示して一覧を再取得（`loadList`）。

**3タブの概要（比較）**

| タブ                          | 何をする                               | 対象項目                                                                        | 送信対象／スキップ                                   | 確認文言                                                  |
| ----------------------------- | -------------------------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------- | --------------------------------------------------------- |
| ① 一括変更（`doBulkChange`）  | 選択した**全日報に同じ変更を一括適用** | チェックON にした項目（日付／案件／作業内容／時間／進捗／遅れ報告／その他報告） | 現在値と異なる日報のみ送信、同値ばかりは「変更なし」 | 「選択した N 件に変更を適用します。よろしいですか？」     |
| ② 個別変更（`doBulkIndiv`）   | **1行ずつ個別編集**                    | 各日報の全項目（行ごと）                                                        | 元の日報から実際に変わった行のみ                     | 「変更した N 件を更新します。よろしいですか？」           |
| ③ 一括置換（`doBulkReplace`） | **テキストの検索置換**                 | 遅れ報告／その他報告                                                            | 実際に置換が発生した日報のみ、変化なしは「対象なし」 | 「選択した N 件のテキストを置換します。よろしいですか？」 |

- **タブ①「一括変更」**（`bulkChangePage` / `doBulkChange`）: **選択した全日報に同じ変更を一括適用**する。変更可能項目は `BULK_CHANGE_FIELDS`＝日付 / 案件（コード＋取引先名＋案件名を一括）/ 作業内容 / 作業時間 / 作業進捗 / 遅れ報告 / その他報告。
  - 各項目を「**変更前｜変更後**」の左右等幅グリッド（`buildBulkChangeGrid`）で表示。**変更前**は選択日報で値が全件同一ならその値を readonly 表示、**値が複数ある項目は「複数あり」ボタン**にして押下で選択日報ごとの当該値を一覧表示（`showBulkMulti`→`bulkMultiModal`）。
  - **変更後**の各項目には**変更対象チェックボックス**（`bulkChecks`）が付き、**チェックON の項目だけを適用対象**とする（変更後コントロールを操作すると自動でチェックON＝`markDirty`）。初期値は先頭選択行（`bulkRows[0]`）の値。日付はカレンダー入力のみ、案件は「検索」で⑤案件モーダル（`openAnkenModal('bulk')`）から設定。
  - 適用時は各日報について、変更後の値が**その日報の現在値と異なる場合のみ**送信し、同値ばかりなら「変更なし」としてスキップ。全項目チェックOFF・チェック項目の入力不備（日報登録と同じ観点＝`bulkChangeError`。進捗↔遅れ報告の相関含む）はトーストで中断。確認ダイアログ（`uiConfirm`「選択した N 件に変更を適用します。よろしいですか？」）承認後に実行。
- **タブ②「個別変更」**（`bulkIndivPage` / `doBulkIndiv`）: 選択日報を**1画面のリストで1行ずつ個別編集**する（`buildBulkIndivGrid`）。日報1件を**多段明細**で並べる（`bi-table`）: **1段目**＝日付 / 案件（`案件コード` readonly ＋「検索」＋案件名表示）/ 作業内容 / 作業時間 / 作業進捗、**2段目**＝遅れ報告（全幅 textarea 5行）、**3段目**＝その他報告（全幅 textarea 5行）。
  - **案件は行ごとに検索**（`openAnkenModal('indiv', e)`。`indivAnkenTarget` に対象行を保持）。案件コードは検索で変わるため初期値（`ankenInitial`）を保持して変更判定に使う。日付はカレンダー入力のみ。
  - **変更検知**（`wireIndivChange`）: 各コントロールは開いた時点の値を初期値として保持し、初期値と異なる間だけ**薄赤（`bi-changed`）**でハイライトして変更を可視化する。
  - 全行を日報登録と同じ観点で検証（`indivRowError`）してから、**元の日報から実際に変わった行のみ**（`isIndivChanged`）を更新対象にする（`buildIndivBody` で `registData` ボディ化）。確認ダイアログ（「変更した N 件を更新します。よろしいですか？」）承認後に実行。
- **タブ③「一括置換」**（`bulkReplacePage` / `doBulkReplace`）: 選択日報の**遅れ報告 / その他報告**のテキストに対し、「置換前」に一致する文字列を「置換後」へ置換（`String.split().join()` による全置換）。**置換前は必須**（空欄では実行不可）、**置換後は空欄可**（一致部分の削除）。実際に置換が発生した日報のみ送信し、変化なしは「対象なし」でスキップ。確認ダイアログ（「選択した N 件のテキストを置換します。よろしいですか？」）承認後に実行。

#### 一括削除（`doBulkDelete`）

- チェックした日報を確認ダイアログの後、`deleteData` で順次一括削除し、成功/失敗件数をトースト表示して一覧を再取得する。
- 下部にステータス行（`outStatus`）と生レスポンス JSON（`<details>`）。**右下に更新ボタン（`btnUpdate`）**＝新しいバージョンがある時のみ表示し、押下で自動更新（§7.2）。
- **担当者の扱い**: メイン（日報一覧）の担当者（`tanto`/`tantoName`）は、初期はログイン担当者（本人自動セット）。**上部バーの「担当変更」ボタン**（§5.0.1）から、参照許可の確認（§5.9）を経て ⑥ 担当者マスタ一覧で選択して切り替えられる（選択後に一覧を取り直す）。担当者コードは readonly で手入力不可。登録モーダルからの担当者変更は**廃止**（登録モーダルの担当者はメイン担当者／編集行の担当者に固定。§5.3）。**「↻ 再読込」でログイン担当者に戻す**: 本人自動セットの結果を `loginTanto` に保持し、再読込時に `restoreLoginTanto` で日報一覧の担当者を `loginTanto` へ戻す（自動セット前は HTML 初期値）。同時に参照のみモードも解除する（§5.9）。

#### 本人の特定（メール捕捉 → `名称4` 照合。v0.2 で刷新）

認証したユーザー本人の担当者コードを、**rkanri ページに表示された担当者コードの直読み（旧 `AUTO_TANTO_PROBE`）ではなく、Microsoft SSO のログインメール ⇒ 担当者マスタ（`getTantoList`）の `名称4` 照合**で確定する。根拠と実機確認は `現行仕様.md §1.1`。

**識別子の対応関係（実機確認済み）**

- Microsoft ログインメールのローカル部（`@` より前。例 `tsuchidas`）＝ 担当者マスタの **`名称4`（英字ログインID）** ＝ desknet's NEO の `UserID`。
- `名称4` は本人に一意（`getTantoList {svalue:''}` 全件に対し 1 件だけ一致 → その `key` が担当者コード）。

**メール捕捉（`main.js` / `webRequest`）**

- rkanri の認証は Microsoft SSO の**認可コードフロー**で、`id_token` はクライアントに渡らない。そのため main プロセスで `session.fromPartition('persist:rkanri').webRequest.onBeforeRequest` を `https://login.microsoftonline.com/common/GetCredentialType*` に絞り、**リクエスト body（JSON）の `username`（＝ログインメール）を捕捉**する。
- **`GetCredentialType`（username のみ）だけを対象**とし、パスワードを含む認証 POST には触れない。捕捉値はメモリ保持し、IPC `get-captured-email` で renderer に渡す。
- `GetCredentialType` は「メール入力 → 次へ」で発火する。**サイレント SSO**（Cookie 保持で無入力ログイン）時は発火しないため、後述の永続化で補完する。

**本人特定フロー（`resolveLoginTanto`。認証成功時＝アプリ遷移前に実行）**

`getTantoList {svalue:''}`（トースト抑制）で担当者マスタ全件を取得し、`名称4`（小文字化）＝照合キー（小文字化）で本人を探す（`lookupTantoByLocal`）。照合キーを ①→②→③ の順で試す。

| 段階                            | 照合キー                                                                                    | ヒット時                                                                                                                                                             | 外れ時                                                                                                                                      |
| ------------------------------- | ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **① SSO 自動照合**              | 今回ログインメールのローカル部（`@` より前。`get-captured-email`。サイレント SSO では無い） | `tanto=key` / `tantoName=名称1`（**マスタ由来**）を確定 → `set-identity` で `{email, code, name, userId}`（`userId`＝ローカル部）を永続化 → アプリへ遷移して一覧表示 | メール無し／不一致 → ②                                                                                                                      |
| **② 永続化ユーザーID で再照合** | 永続化済み `identity.json` の `userId`（前回 SSO／手動で確定した値）                        | 確定・再永続化 → アプリへ遷移。＝**過去に一度確定した本人は、以後メールが取れなくても（サイレント SSO でも）`userId` で自動認証される**                              | `名称4` 変更等で不一致 → ③                                                                                                                  |
| **③ ユーザーID 手動入力へ誘導** | （自動照合を断念）                                                                          | —                                                                                                                                                                    | 認証画面に留まり**手動入力パネル**（`showManualAuth`。§5.1）を表示（`{ error:'manual', email }`。捕捉メールがあれば入力欄の初期候補にする） |

- **強制再認証（純初回のみ）**: メールも永続情報も**どちらも無い**純初回だけ、③の前に**一度だけ**強制再認証（`{ error:'reauth' }`→`reauth()`＝`clearSession` → webview 再読込）して対話ログインを促し、メールを捕捉し直す（`forcedReauthTried` で1回に制限）。

**手動入力による確定（第2フェーズ。`submitManualId`）**

- 手動入力パネルで入力したユーザーID を照合キーに `名称4` を検索（`lookupTantoByLocal`）。
  - **ヒット**: 入力ユーザーID を含めて `{email, code, name, userId}` を永続化（→ 次回以降は上記②で自動認証される）し、アプリへ遷移。
  - **ヒットなし**: エラー「担当者情報が存在しないため、ログインできません。<br>上長か、管理者にお問い合わせください。」を表示（`<br>` を含むため `innerHTML` で描画）。入力欄は残して再入力可能。
- ブラウザ表示版は永続化を main 側で行うため `POST /api/manual-login` に委譲する（§11.4）。

**永続化（`main.js`。`userData/identity.json`）**

- `get-identity` / `set-identity` / `clear-identity` を IPC で提供。本人を確定できた時点で `{email, code, name, userId}` を保存する（旧仕様の `{email, code, name}` に **`userId`（`名称4` 照合キー）を追加**。②の自動再照合に用いる）。
- **再認証（`clear-session`）時に捕捉メールと `identity.json` を破棄**し、次回ログインで本人を取り直す（別アカウントへの切替に対応）。

**担当者コードの直読み廃止**

- 旧 `AUTO_TANTO_PROBE`（rkanri ページの `<input>` から `/^P-\d{6}$/` を拾う方式）と HTML 初期値 `P-000184` 依存は廃止。本人特定は上記フローに一本化する。
- 「担当変更」（§5.9）で他担当者を選ぶ機能、`loginTanto` による「↻ 再読込」での本人復帰（§5.0.1 / `restoreLoginTanto`）は従来どおり維持する。`loginTanto` は本フローで確定した本人（`{code, name}`）を保持する。

  - 取引先名(`torihikisakiMei`, readonly) / 案件名(`ankenMei`, readonly)
  - 作業内容(`sagyoNaiyo`, select) / 作業時間(`sagyozikan`) / 作業進捗(`sagyoShinchoku`, select)

    - **登録ボタンの緑化**: コピー登録中は登録ボタン(`btnRegist`)を緑（`.ok`）にする（§5.3 フッタ）。
    - **浮き上げアニメーション**: 修正モードからコピーモードへ切り替える際、**先にモーダル（`#registModal .modal`）を少し上へ浮き上げてから元の位置に戻す**演出を見せ、**元の位置に戻ったタイミング（`animationend`）でコピーモードへ切り替える**（`playCopyHop` が `.copy-hop` クラスを一時付与＝`@keyframes copy-hop`。`animationend` でクラスを除去し、コールバックで `enterCopyMode`→`updateMode` を実行）。＝アニメーション中は修正モードの表示のまま、着地と同時にコピーモード表示（緑バッジ・案内パネル・緑の登録ボタン）へ変わる。コピーは修正モードのみ押下できるため、常にこの演出が入る。
  - **リセット**（`resetForm`）: **通常時**はモーダルを**開いた時点の状態に戻す**（新規＝デフォルト表示、編集＝選択された未編集の値）。開いた時に `takeSnapshot()` で全項目（`SNAPSHOT_FIELDS`＝送信10項目 `REG_FIELDS` ＋表示担当者 `regTanto`/`regTantoName`）を記録し、復元して合計も取り直す。**コピー登録中**は新規モード（`nippoCd` 空）のまま入力欄を**コピー元＝スナップショットの値へ戻す**（`nippoCd` は復元せず、コピー登録モードは維持）。
  - **戻る**（`btnBack` / `backToEdit`、コピー登録中のみ表示）: **コピー押下前の修正モードの初期表示状態に戻る**（`nippoCd` も復元して修正モードへ戻し、コピー登録モードを解除する）。
- 日付ボタンは現在日付 +offset を設定する。**画面内部は `yyyy-mm-dd`（`type="date"` の形式）で保持し、API 送信時に `yyyy/m/d`（ゼロ埋めなし）へ変換**する（`toApiDate`）。修正モードで一覧の `yyyy/m/d` を読み込む際は `toInputDate` で `yyyy-mm-dd` へ変換して入力欄へ設定する。
- **合計は自動取得**: 登録モーダルを開いた時（新規＝当日／修正＝その日報の日付）に `getTotal` を自動実行して `totalVal` に表示する（`doTotal`、トースト抑制）。日付ボタン押下・日付手入力（`change`）でも再取得する。**「合計取得」ボタンは廃止**（合計値ラベルのみ表示）。
- `tanto`/`tantoName` の正はメイン画面側。登録モーダルの `regTanto`/`regTantoName` は表示ミラーで、新規時に `syncRegTanto` でメイン担当者を複製、修正時は行の担当者。送信 `tanto` は `regTanto` の値を使う（＝表示中の担当者）。

### 5.4 ⑤ 案件マスタ検索モーダル（`ankenModal`）

- 検索テキスト(`svalue`) + 「検索」(`gethistory {tanto, svalue}`) / 「履歴」(`gethistory {tanto}`)。
- **開いた時に自動で「履歴」(`gethistory {tanto}`)を実行して初期表示**する（`svalue` はクリア）。「検索」ボタンは再検索用。
- 結果テーブル（`ankenTable`）を表示。行クリック(`pickAnken`)で **`ankenCd`/`torihikisakiMei`/`ankenMei`** を登録モーダルへ設定し、案件モーダルを閉じる。

### 5.5 ⑥ 担当者マスタ検索モーダル（`tantoModal`）

- 検索テキスト(`svalue2`) + 「検索」(`getTantoList {svalue}`)。
- **開いた時に自動で全件(`getTantoList {svalue:""}`)を実行して初期表示**する（`svalue2` はクリア）。「検索」ボタンは再検索用。
- **開く経路は上部バーの「担当変更」のみ**（`openTantoModal`。参照許可の確認 §5.9 を経て開く。登録モーダルからは開かない＝①）。
- 結果テーブル（`tantoTable`）を表示。行クリック(`pickTanto`)で **メイン（日報一覧）の `tanto`/`tantoName`** を選択行（`key`/`名称1`）に設定し、直前の許可確認で選んだモード（全権/参照のみ）を `applyViewOnly` で反映して、モーダルを閉じて**一覧を取り直す**（`loadList`）。

### 5.6 テーブル表示の共通仕様

`renderTableInto(container, endpoint, data, onRowClick, selectable)` で任意コンテナへ描画する（メイン一覧・各モーダル一覧で共用。`selectable=true` で先頭に選択チェックボックス列を付ける＝日報一覧用）。

- **表示列の絞り込み**: エンドポイント別ホワイトリスト（`DISPLAY_COLS`）で該当列のみ表示。`getNippoList` は実測25フィールド返るため（[現行仕様.md](現行仕様.md) §7.4）、日付 / 案件コード / 取引先名 / 案件名 / 作業時間 / 作業内容 / 作業進捗 / その他報告事項 に絞る。未定義／非該当は全列にフォールバック。
- **見出しラベル**: `HEADER_LABELS` で日本語化。`getTantoList` の `key`→`担当者コード` / `名称1`→`担当者名`（同 §6.2）。
- **非配列応答は既存テーブルを維持**: `getTotal`（スカラー）・`registData`/`deleteData`（`{status}`）は再描画せず直前の表示を残す。
- **空配列 `[]`（該当0件）**: **テーブルヘッダーは維持したまま**、ボディに「**該当データがありません**」を全列結合（`colSpan`）した1行で表示する。**このときテーブルを一覧領域いっぱいに広げ（`table.is-empty`＝`height:100%`）、セルを縦中央（`.empty-cell`）にして、帯状ではなくボディ全体の中央に見せる**。空データ時の列はホワイトリスト（`DISPLAY_COLS`）からヘッダーを構成する。ホワイトリスト未定義でヘッダー列を特定できない場合のみ、ヘッダーなしで領域中央にメッセージ表示（`.empty-list`）へフォールバック。
- **列幅とセルのはみ出し**:
  - `renderTableInto` は各セルにエンドポイント別クラス（`tbl-<endpoint>`）と列名（`data-col`）を付与し、**セル内容は内箱（`div.cell`）に入れる**。
  - 日報一覧（`tbl-getNippoList`）は **選択列（`.chk`）/ 日付 / 案件コード / 取引先名 / 案件名 / 作業時間 / 作業内容 / 作業進捗 を固定幅**（`width`/`min`/`max` 一致）で表示し、残り列（その他報告事項）が可変幅で残りを埋める。
  - 固定幅を超える文字は右端で見切れさせ（`…` 省略はしない）、**内箱を横スクロールコンテナ（スクロールバー非表示）にして、カーソルの範囲選択で見切れた部分をたどり確認・コピーできる**（`td` 自体は `table-cell` でスクロールできないため内箱を用いる）。
  - 行クリックは編集モーダルを開くが、**セル内テキストを範囲選択した直後のクリックでは編集を開かない**（選択・コピー操作を妨げないため）。
- **行データは全フィールド保持**。表示列を絞っても行クリック時の反映（`日報コード`・`遅れ報告` 等の非表示列を含む）に影響しない。

### 5.9 担当変更と参照許可・参照のみモード

上部バーの「担当変更」ボタン（`btnTantoChange`。§5.0.1）で他担当者の日報を閲覧・操作する際、**事前に本人から参照許可を得ているか**を確認し、権限レベルを選ばせる。

- **フロー**: 「担当変更」押下 → ⑥ 担当者マスタ一覧を開く → 担当者を選択（`pickTanto`）→ 選択した担当者を `pendingTantoRow` に保持し、参照許可の確認モーダル（`permModal`）を表示。
  - **ログイン担当者と同じ担当者を選んだ場合は確認を出さず**、そのまま全権で変更する（`loginTanto.code` と一致判定）。
- **参照許可の確認モーダル（`permModal`）**: メッセージは選択した担当者名を差し込み「**[担当者名]さんから、事前に参照の許可を得ていますか？**」（`permMsg` に動的設定）。ボタンは **[全権許可]**（`btnPermFull`）/ **[参照のみ]**（`btnPermView`）。
  - **[全権許可]**（`confirmPerm(false)`）: 担当者を変更（登録・更新・削除まで可能）。
  - **[参照のみ]**（`confirmPerm(true)`）: 担当者を変更したうえで**参照のみモード**にする。
  - **✕**（`cancelPerm`）: 担当変更を中断（担当者は変更しない。`pendingTantoRow` を破棄）。※全モーダル共通で背景（外側）クリックでは閉じない方針のため、中断は ✕ のみ。
- **確定処理（`applyTantoChange`）**: `tanto`/`tantoName` を選択行（`key`/`名称1`）に設定 → `applyViewOnly(view)` でモード反映 → 一覧を取り直す（`loadList`）。`applyViewOnly` は `body.view-only` クラスを切り替え、状態は `viewOnly` フラグで保持。
- **参照のみモードで非表示になる編集系 UI**（`body.view-only` の CSS で制御）:
  - 日報一覧: **選択列（チェックボックス）** / **新規登録** / **一括コピー** / **一括変更** / **一括削除**。
  - 登録モーダル: **フッタ全体**（登録/更新・削除・コピー・リセット・戻る）。
- **閲覧は可能**: 一覧の行クリックで登録モーダルは開ける（詳細確認）。その際バッジは「修正モード」ではなく **「参照モード」**（`badge view`）で表示する（`updateMode` が `viewOnly && isEdit` で判定）。
- **多重防御**: UI 非表示に加え、`doRegist`/`doDelete`/`openNewModal`/`openBulkCopy`/`doBulkDelete`/`openBulkEdit`/`doBulkChange`/`doBulkIndiv`/`doBulkReplace` は `viewOnly` 時にトーストで中断し、実行しない。
- **モード解除**: 「↻ 再読込」でログイン担当者に戻す時（`restoreLoginTanto`）と、ログイン直後（`enterApp`）に `applyViewOnly(false)` で解除する。

| UI 操作      | endpoint       | 送信 body                                   |
| ------------ | -------------- | ------------------------------------------- |

| 合計取得     | `getTotal`     | `{ tanto, date }`                           |
| 登録 / 更新  | `registData`   | 全11項目（`nippoCd` 空=新規 / 値あり=更新） |
| 削除         | `deleteData`   | `{ nippoCd }`                               |
| 案件検索     | `gethistory`   | `{ tanto, svalue }`                         |

### 6.1 登録・更新の送信項目（全11）

`nippoCd, tanto, date, ankenCd, torihikisakiMei, ankenMei, sagyoNaiyo, sagyozikan, sagyoShinchoku, okureHokoku, sonotaHokoku`

### 6.2 業務ロジック（現行準拠）

- **モード判定**: `nippoCd` が空=新規モード、値あり=修正モード（バッジ表示）。
- **コピー**（登録モーダル）: 修正モードで押下すると**コピー登録モードへ切替**し、`nippoCd` を空に（他項目は保持）→ 新規化。**コピー元パネルは廃止**し、コピー後も日報登録モーダルだけを表示する。コピー登録中の **リセット**＝コピー元の値に戻す（新規のまま）、**戻る**＝コピー前の修正モードへ復帰（§5.3）。
- **登録/更新前の確認ダイアログ**（`uiConfirm`。§5.9 の native confirm 不使用に準拠）:
  - **修正モード**（`nippoCd` あり・非コピー）で更新押下時 → **「修正します。よろしいですか。」**。
  - **コピーモード**で登録押下時 → **「日報コード {コピー元の日報コード} をもとにコピー登録します。よろしいですか。」**（コピー元の日報コードは `formSnapshot.nippoCd`）。
  - 新規モード（コピー以外）は確認なしでそのまま登録。いずれもキャンセルで中断。
- **登録/更新成功後**: フォーム初期化 → `getNippoList` で一覧再取得。
- **削除**: `nippoCd` 空なら中断。確認ダイアログ（「削除します。元に戻せませんのでご注意を」）承認後に実行 → 初期化 → 一覧再取得。

#### 成否判定（`status` ベース）

`registData` / `deleteData` は HTTP 200 でもボディ `{"status":"success"}` を返す（[現行仕様.md](現行仕様.md) §8.3/§8.4）。本アプリは現行画面でコメントアウトされていた `status` 判定を**実装として有効化**する。

- `isBizOk(r)` = HTTP OK **かつ**（応答が `{status:...}` 形式なら `status === "success"`）。
- `isBizOk` が真のときのみ「成功」toast → フォーム初期化 → 一覧再取得を実行。
- HTTP 200 でも `status !== "success"` の場合は**失敗扱い**（フォーム保持・一覧再取得なし）。
- **エラー時はサーバー応答をそのまま表示する**（canned な日本語メッセージに変換しない）。エラー時の応答形（`status` 値や構造）が未確認のため、`rawText()` で応答本体（オブジェクトは JSON 文字列化／文字列はそのまま）をステータス欄と toast に出力する。生レスポンスは従来どおり `<details>` にも表示。
- 一覧系（`getNippoList`/`gethistory`/`getTantoList`＝配列）・`getTotal`（スカラー）は `{status}` を持たないため、従来どおり HTTP OK で成功扱い。HTTP エラー（非 2xx）時も応答本体をそのまま表示。

### 6.3 入力バリデーション（`checkError` 相当）

上から評価し、最初に該当したものを toast 表示して中断。

1. `regTanto` 空 → 担当者が未入力です
2. `ankenCd` 空 → 案件が未入力です
3. `date` 空 → 日付が未入力です
4. `date` 不正（`new Date` が NaN） → 日付が不正です
5. `sagyoNaiyo` 空 → 作業内容が未入力です
6. `sagyozikan` 空 → 作業時間が未入力です
7. `sagyozikan` 非数値 → 作業時間が不正です
8. `sagyoShinchoku` 空 → 作業進捗が未入力です
9. `sonotaHokoku` 空 → その他報告事項が未入力です
10. `sagyoShinchoku` が「順調」以外かつ `okureHokoku` 空 → 作業進捗が順調以外の場合は遅れ報告を入力してください

### 7.1 単一 exe（electron-builder）に関する既知の制約

- `npm run dist` / `npm run publish`（いずれも `electron-builder --win nsis`）は署名ツール（winCodeSign）展開時に、アーカイブ内の macOS 用 `.dylib` シンボリックリンクを作ろうとし、**Windows のシンボリックリンク作成権限**（管理者 or 開発者モード）が無いと `7za` が「必要な特権を保有していません」で失敗することがある（この `.dylib` は Windows ビルドには不要）。
- **本環境では回避済み**: winCodeSign アーカイブを、権限不要で正しいキャッシュフォルダ名（`%LOCALAPPDATA%\electron-builder\Cache\winCodeSign\winCodeSign-2.6.0`）へ手動展開（macOS の 2 エラーは無視、Windows 用ツールは全て展開される）してキャッシュを用意した。以降は **Developer Mode なしでビルド可能**。
- 上記キャッシュを削除すると再発する。その場合は手動展開をやり直すか、開発者モード ON／管理者権限ターミナルでビルドする。

### 7.2 自動更新（GitHub Releases / electron-updater）

**ログイン画面（起動時）と日報一覧（初期表示時）の両方**で最新バージョンを確認し、更新があれば通知・適用できる。**配信手順は [リリース手順.md](リリース手順.md) を参照**。

- **配置元**: GitHub Releases（`build.publish` = provider `github` / owner `ProGTST` / repo `NippoHokoku`）。electron-builder が `latest.yml` と NSIS インストーラを Release に発行し、`electron-updater` がそれを見て更新有無を判定する。
- **発行手順（リリース時）**:
  1. `package.json` の `version` を上げる（例 `0.1.0` → `0.1.1`）。
  2. `GH_TOKEN`（repo 権限のトークン）を環境変数に設定。
  3. `npm run publish` を実行（NSIS ビルド → GitHub Release へ `latest.yml` と `日報入力-Setup-<version>.exe` をアップロード）。Release を公開する。
  - ⚠ **リポジトリが private の場合**、クライアント実行時にも `GH_TOKEN` が必要（配布端末に埋め込むのは非推奨）。運用上は **public リポジトリ**か、社内 HTTP の generic provider への切替を検討。
- **メイン側（`main.js`）**: `electron-updater` は require 時に `app` へ触れるため **遅延生成（`getUpdater()`）** で扱う。`autoDownload=false`（ボタン/確認で明示ダウンロード）。IPC:
  - `check-update`: `checkForUpdates()` → `{ available, version, current }`。**未パッケージ（開発）時は `available:false`**（`app.isPackaged` で判定）。
  - `start-update`: `downloadUpdate()`。進捗・完了は `update-status` イベントでレンダラーへ通知。
  - `quit-and-install`: `quitAndInstall()` で再起動して適用。
  - 状態通知（`webContents.send('update-status', …)`）: `available` / `none` / `downloading{percent}` / `downloaded` / `error`。レンダラー側の購読（`onUpdateStatus`）は進捗表示に必要な `downloading` / `downloaded` / `error` のみ処理する（`available` / `none` はボタン表示を `check-update` の戻り値で制御するため購読側では未処理）。
- **プリロード（`preload.js`）**: `appApi.checkUpdate()` / `startUpdate()` / `quitAndInstall()` / `onUpdateStatus(cb)` を公開。
- **UI（`renderer.js`）**: 更新ボタンと現在バージョンを **ログイン画面フッター右端（`btnUpdateLogin` / `verLabelLogin`）** と **日報一覧フッター右下（`btnUpdate` / `verLabel`）** の両方に配置。並びはいずれも **[更新ボタン] [現在のバージョン]**。両画面のボタン・ラベルは `updateBtns()` / `setVersionLabel()` でまとめて同期し、状態（表示/非表示・進捗 `⬆ NN%`・現在バージョン）は常に一致させる。
  - **現在のバージョンは常時表示**（`v<version>`）。初期は `appInfo.version`、確定は `check-update` の `current`（＝`app.getVersion()`）で上書き。
  - **起動時（ログイン画面）に `checkForUpdate` を実行**し、更新があればログインフッターに更新ボタンを表示。**日報一覧の初期表示時（`enterApp` → `checkForUpdate`）**にも再チェックする。確認ダイアログの多重表示は `updatePromptShown`（`showAuth` でリセット）で1セッション1回に制限。
  - **最新と一致（または開発時・チェック失敗）→ 更新ボタン非表示**。**更新あり → 更新ボタンを常設表示**し、確認ダイアログ「最新のバージョン `{version}` がアップロードされています。（改行）最新のバージョンにアップデートしますか？」を表示（`confirm` はネイティブのため改行は `\n`）。
    - **OK**: そのまま更新（ダウンロード → 完了で再起動・適用）。
    - **キャンセル**: 更新せず通常利用を継続。**更新ボタンは残す**ので、任意のタイミングで押して更新できる。
  - 更新ボタン押下（`doUpdate`）: 再確認後に `startUpdateFlow`（`start-update`）。進捗はボタンに `⬆ NN%` 表示。`downloaded` で `quitAndInstall`。

## 8. セキュリティ・運用上の留意点

- **認証は本人のブラウザセッションに依存**。アプリは本人の権限・データ範囲でのみ動作する。認証情報をアプリが保存・送信することはない（Cookie は webview の `persist:rkanri` パーティションが保持）。
- **XSRF トークンはハードコードしない**。実行時に Cookie から都度取得する。
- 社内ネットワーク（LAN/VPN）接続時のみ対象ページに到達できる前提。オフライン/社外では webview 読込が失敗し、接続ステータスが赤表示になる。
- `registData` はサーバー応答の `status` によるエラー判定が現行画面ではコメントアウトされている（[現行仕様.md](現行仕様.md) §8.3）。本アプリでは HTTP ステータスと本文で成否を判定しているが、業務エラー（重複・制限超過等）の詳細レスポンス仕様は要確認。

## 11. ブラウザ表示モード（トレイ常駐＋ローカル proxy。実装済み）

デスクトップアプリと同じ画面を、**普通のブラウザのタブで** `http://nippo.local` から表示できるようにする拡張。exe をデスクトップ起動する従来動作は非改変で維持し、両立させる。**本節は `main.js`（トレイ常駐・ローカル HTTP サーバー・proxy・本人特定・hosts 昇格）／`renderer.js`（`IS_BROWSER` 分岐）に実装済み**。

### 11.1 なぜローカル proxy が必要か

ブラウザのタブから `rkanri.genech.co.jp` の API を直接叩くことはできない（別オリジン＝CORS／Cookie(SameSite) で遮断）。デスクトップ版はこれを `<webview>.executeJavaScript`（rkanri オリジンでの fetch）で回避している（§4）。ブラウザ版では代わりに、**常駐アプリ（Electron main）が保持する `persist:rkanri` セッションを使って rkanri API を代理呼び出しするローカル proxy** を経由する。

```
ブラウザ(http://nippo.local)
  │  ①UI(HTML/JS/CSS) を取得   ②/api/* を呼ぶ（同一オリジン＝CORSなし）
  ▼
常駐アプリ内 ローカルHTTPサーバー（127.0.0.1）
  │  proxy: persist:rkanri の Cookie/XSRF を付与して転送（Electron net）
  ▼
rkanri.genech.co.jp/kanri/nippo/*
```

**proxy の実体（`proxyToRkanri`）**: `persist:rkanri` セッションの Cookie から `XSRF-TOKEN` を取り出して `x-xsrf-token`（`decodeURIComponent` 済み）に付与し、`net.request({ method:'POST', url: RK_ORIGIN + '/kanri/nippo/' + endpoint, session, useSessionCookies:true })` で中継。応答本文を読み切って `{ ok, status, body }` を返す（renderer へは `{ ok, status, data }` に整形。JSON パース不能な HTML/login 応答は文字列のまま渡し、renderer 側で未ログイン検出）。

**SSO ログインウィンドウ（`openLoginWindow`）**: `webPreferences.partition = persist:rkanri` の `BrowserWindow`（520×720）で rkanri を開き、`did-stop-loading` ごとに `getTantoList` が配列を返せば（＝ログイン成立）ウィンドウを閉じて成功を返す。ユーザーが閉じたら中断（`false`）、**3分の安全弁**（`setTimeout`）で強制終了する。

### 11.2 常駐（トレイ）

- Windows ログイン時に**トレイ常駐**で自動起動（`app.setLoginItemSettings({ openAtLogin, args:['--tray'] })`。`--tray` 起動ではウィンドウを出さずバックグラウンド常駐）。
- トレイ右クリックメニュー:
  - **ブラウザで開く**
  - デスクトップ画面を開く
  - ブラウザ表示を有効化（hosts 追記）
  - 終了
- 「ブラウザで開く」は既定ブラウザで URL（`browserUrl`）を開く（hosts に nippo.local があれば `nippo.local`、無ければ `127.0.0.1`）。**実際に確保したポートを URL に反映**し、80 番以外（例 8090）で待ち受けている場合は `:ポート`（例 `http://nippo.local:8090/`）を付与する（暗黙の 80 番＝競合相手に繋がるのを防ぐ）。サーバーが確保できていない場合は開かず通知に留める。
- ブラウザ利用時に見えるのは**ブラウザのみ**。デスクトップウィンドウは exe 起動時か SSO ログイン時のみ表示。
- 二重起動は `requestSingleInstanceLock` で抑止し、2 回目の起動はデスクトップ画面を出して終了する。ウィンドウを全て閉じても常駐は維持し、終了はトレイの「終了」から行う。

### 11.3 URL（`http://nippo.local`。前提。非管理者は `http://127.0.0.1`）

- **前提 URL は `http://nippo.local`**。`nippo.local` を 127.0.0.1 に解決させるため、**hosts に `127.0.0.1 nippo.local` を1行追加**する。
- 追加はインストール／アップデート時に自動試行するが（バージョン単位で一度）、**hosts 編集には管理者権限（UAC 昇格）が必要**。現行インストーラはユーザー単位（管理者不要）のため、**この行の追加だけを昇格ヘルパー（PowerShell `Start-Process -Verb RunAs`）で実行**する（インストーラ全体の権限レベルは変えない）。
- **hosts 追記の判定は厳密**: 行内コメントを除去し、先頭が `127.0.0.1` かつ `nippo.local` をホスト名トークンに含む**非コメント行のみ**を「存在」とみなす（コメントアウト行は存在扱いにしない／`nippo.localhost` 等は誤検出しない）。JS 側（`hostsHasEntry`）と昇格 PowerShell 側（`Select-String` の正規表現）の両方で判定する。
- **昇格ヘルパーの実装**（`ensureHostsEntry`）: `userData/add-nippo-hosts.ps1` を書き出し、`powershell.exe … Start-Process powershell.exe -Verb RunAs -WindowStyle Hidden -File <ps1>` で **UAC 昇格して実行**する（追記行は `127.0.0.1 nippo.local  # nippo-desktop-app …`）。
  - `.ps1` は **BOM 無し UTF-8** で書き出すため、埋め込むコメントは **ASCII のみ**にする（日本語を混ぜると PowerShell 5.1 が ANSI と誤解して文字化けする）。
  - `.ps1` パスを PowerShell のシングルクォート文字列へ埋め込む際、`'` を `''` でエスケープする（Windows ユーザー名に `'` が含まれても壊れず、注入もされない）。
  - 昇格拒否・失敗は `spawn(...).on('error')` で握りつぶし、**ブラウザ表示が無効になるだけ**で許容する（`127.0.0.1` フォールバックは有効）。
- **自動試行はバージョン単位で一度**（`ensureHostsEntryOnce`）: 未パッケージ（開発）時は自動昇格しない。既に有効行があれば何もしない（UAC を出さない）。マーカー `userData/.hosts-attempted-<version>` の有無で「同バージョンで試行済み」を判定し、**新バージョン起動時に一度だけ**再試行する。
- **昇格が拒否された／管理者権限が無い場合**でも、アプリは `127.0.0.1:80` で待ち受けるため **`http://127.0.0.1/` でブラウザ表示は可能**（非管理者環境のフォールバック。実機確認済み）。後から手動または管理者が hosts を追記すれば `nippo.local` も有効化される。
- ポートは **`[80, 8090]` を先頭から順に試し、確保できたものを使う**（`LOCAL_PORTS`）。80 が Docker/WSL 等（Go 製 HTTP サーバー等）に奪われている環境では **8090 にフォールバック**する（`EADDRINUSE` を `server.on('error')` で受けて次候補を `listen` し直す）。確保できたポートは `activePort` に記録し、`browserUrl` の `:ポート` 付与に反映する。
- **全候補が使用中**の場合はブラウザ表示を無効化（`activePort=null`）し、`[nippo] local server error` をログ出力するとともに、**デスクトップ通知**（`notifyBrowserUnavailable`）で「ブラウザ表示は利用できません」をユーザーへ知らせる（無言失敗を避ける）。デスクトップ版はそのまま通常動作。
- ⚠ `localhost` も許可ホスト（`ALLOWED_HOSTS`）に含むが、`localhost` を別システムで使用している環境がある。運用は `nippo.local`／`127.0.0.1` を用いる。

### 11.4 ログイン（ブラウザ起点 → SSO はアプリ窓）

- proxy が使う Cookie は Electron の `persist:rkanri` に入っている必要がある。ブラウザのタブで rkanri にログインしても Cookie はブラウザ側に入り proxy からは使えないため、**SSO ログイン画面はアプリ側のウィンドウで表示**する。
- 流れ: ブラウザで `nippo.local` を開く → **起動時に `GET /api/whoami`**（`browserBoot`）で本人を確認 → 未ログインなら「ログイン」ボタン表示 → 押下で **`POST /api/login`**（`browserLogin`）→ 常駐アプリが **SSO ログイン用ウィンドウ（`persist:rkanri`）をポップアップ**（`openLoginWindow`）→ Microsoft ログイン（§本人特定のメール捕捉・名称4 照合もここで動作）→ Cookie 確立後にポップアップを閉じ、本人を返してブラウザ画面が利用可能になる。
- ＝「操作の起点はブラウザ、SSO 入力欄だけアプリ窓」。構造上、SSO 入力をブラウザタブ内に置くことはできない。
- **本人特定はデスクトップ版と同一ロジックを main 側に実装**（`resolveWhoami`＝①捕捉メール→名称4 → ②永続化 `userId`→名称4 → ③手動入力へ誘導。§5.2）。自動照合できない場合は画面に**手動入力パネル**を出し、**`POST /api/manual-login`**（`handleManualLogin`）でユーザーID を確定・永続化する。
- **ブラウザ用ローカル API エンドポイント**（`/api/*`。トークン Cookie＋Origin 検証を通過したもののみ。§11.6）:

  | メソッド | パス                | 機能                                                                                         |
  | -------- | ------------------- | -------------------------------------------------------------------------------------------- |
  | `GET`    | `/api/whoami`       | 本人確認（未ログイン `{error:'login'}` / 照合不可 `{error:'manual'}` / 確定 `{code, name}`） |
  | `POST`   | `/api/login`        | SSO ポップアップを開いてログイン→本人特定                                                    |
  | `POST`   | `/api/manual-login` | 手動入力ユーザーID で本人特定・永続化                                                        |
  | `POST`   | `/api/logout`       | セッション・捕捉メール・`identity.json` を破棄（再認証で使用。§4.4）                         |
  | `POST`   | `/api/<endpoint>`   | rkanri API の proxy（`API_WHITELIST` の6種のみ）                                             |

### 11.5 renderer の 2 モード化

- **実行環境の判定**: `IS_BROWSER = !window.appApi`（preload の有無）。起動時に `IS_BROWSER` なら `document.documentElement` に `browser-mode` クラスを付け（CSS で webview 関連 UI を隠しログインボタンを出す）、`browserBoot()` を実行。デスクトップ版は従来どおり `showAuth()`＋更新チェック。UI（画面・操作）本体は共通化する。
- **API 実行層の抽象化（`execApi`）**: `callApi` の下層で実行環境を吸収し、いずれも `{ ok, status, data }` 形を返す。
  - **Electron webview 内**（デスクトップ版）: `rk.executeJavaScript(buildInjection(...))`＝対象ページのオリジンで same-origin fetch（§4）。
  - **ブラウザ内**（ブラウザ版）: 同一オリジンの `POST /api/<endpoint>` を fetch（body は現行と同じ JSON）。ローカルサーバーが proxy 結果を `{ ok, status, data }` に整形して返す（§11.1）。
- **共通の後処理（`callApi`）は両モードで一本化**: ログイン切れ検出（`data` が文字列で `<html`/`<!doctype`/`login` を含む→認証画面へ戻す。トーストの文言のみ「ログイン」導線に合わせて出し分け）、成否判定（`isBizOk`／`status !== "success"` は業務エラー）、生レスポンス JSON 表示、成功/エラーのトーストは §4.2／§6.2 と同一。
- **ブラウザ版の認証・制御 fetch**（`apiGet`/`apiPost`。JSON を返す薄いラッパー）: 起動時 `GET /api/whoami`（`browserBoot`）、ログイン `POST /api/login`（`browserLogin`）、手動 `POST /api/manual-login`（`submitManualId`）、ログアウト `POST /api/logout`（`reauth`）。結果は `browserHandleWhoami` が「確定→一覧／`manual`→手動入力パネル／`login`→ログイン待ち」に振り分ける。

### 11.6 セキュリティ（必須）

ローカル proxy は「認証済み rkanri へのゲートウェイ」になるため、同一 PC 上の他プロセス/ページからの悪用を防ぐ。多層で防御する。

- **127.0.0.1 限定バインド**: `httpServer.listen(port, '127.0.0.1')`。外部 NIC では待ち受けない。
- **Host ヘッダ検証**（`hostOf`）: すべてのリクエストで `Host` のホスト名が `ALLOWED_HOSTS`（`nippo.local` / `127.0.0.1` / `localhost`）に含まれなければ **403**（DNS リバインディング対策）。
- **起動ごとのランダムトークン**（`serverToken`＝`crypto.randomBytes(24)` の hex）: **UI（`index.html`）配信時に `Set-Cookie: nippo_token=<token>; HttpOnly; SameSite=Strict; Path=/` で埋め込む**。`/api/*` は `apiAuthorized` で **Cookie の `nippo_token` がサーバー保持値と一致**することを必須にする。`HttpOnly` によりページ JS からトークンを読めず、`SameSite=Strict` で他サイトからの送信を防ぐ。トークンは起動ごとに変わるため、旧トークンでの再利用も不可。
- **Origin チェック**（`apiAuthorized`）: `Origin` ヘッダがある場合はそのホスト名も `ALLOWED_HOSTS` に限定する。
- **proxy 対象の限定**（`API_WHITELIST`）: 中継してよいのは `getNippoList` / `getTotal` / `gethistory` / `getTantoList` / `registData` / `deleteData` の**6エンドポイントのみ**。任意 URL 転送は不可（`/kanri/nippo/<endpoint>` に固定連結）。
- **静的配信は固定ファイルのみ**: `src/` 配下の `index.html` / `renderer.js` / `styles.css` / `icon.png` のみを既知パスで配信し、パス結合による外部ファイル読み出しはしない。
- **XSRF トークン**は従来どおりハードコードせず、`persist:rkanri` の `XSRF-TOKEN` Cookie を都度取得（`decodeURIComponent`）して `x-xsrf-token` に付与する（`proxyToRkanri`）。rkanri へは `net.request({ session, useSessionCookies: true })` で `persist:rkanri` の Cookie ごと送出する。

### 11.7 スコープ / 影響

- 追加: main（HTTP サーバー＋proxy／トレイ常駐・自動起動／SSO ポップアップ制御／hosts 昇格ヘルパー）、renderer（API 層の 2 モード化）、セキュリティ層。
- 非改変: exe ダブルクリックによるデスクトップ動作、既存の各画面・API 操作。
- ⚠ 常駐プロセスは Electron のメモリを常時消費する。必要に応じ「一定時間アクセスが無ければ自動終了」等の制御も検討可（今回はトレイ常駐で確定）。
