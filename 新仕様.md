# 新仕様（開発アプリ仕様書）

日報登録デスクトップアプリ（案A: WebView 組み込み / Electron）の仕様・実装をまとめる。
対象システムの既存仕様は [現行仕様.md](現行仕様.md)、プロジェクト背景・要望は [README.md](README.md) を参照。

---

## 1. 目的・方針

- 社内 LAN / VPN 接続端末で、**exe を実行して起動するデスクトップアプリ**から、日報登録画面（`https://rkanri.genech.co.jp/kanri/nippo`）の各 API を実行できるようにする。
- 既存 Web 画面を直接操作せず、アプリ独自の GUI から登録・更新・削除・一覧取得・合計取得・マスタ検索を行う。

### 採用アーキテクチャ（案A: WebView 組み込み）

対象 API は `credentials: include`（Cookie セッション）と**セッション毎に変わる XSRF トークン**を要求するため、外部プロセスからの直接呼び出しは認証の取り回しが難しい。
そこで**アプリ内に対象ページを読み込む WebView を持ち、その（ログイン済み）オリジンのコンテキストで fetch を実行する**方式を採る。これにより Cookie / XSRF が自動的に有効な状態で API を呼べる。

```
┌───────────────────────────── Electron BrowserWindow ─────────────────────────────┐
│  自前UI（src/index.html + renderer.js）        <webview src="…/kanri/nippo">       │
│  ┌───────────────────────────┐                ┌──────────────────────────────┐   │
│  │ 操作パネル（入力・ボタン） │  executeJavaScript(fetch…)                     │   │
│  │ 結果テーブル / JSON        │ ─────────────▶ │ rkanri のページ（ログイン済） │   │
│  │                            │ ◀───────────── │ same-origin fetch + Cookie +  │   │
│  └───────────────────────────┘   結果(JSON)    │ XSRFトークン                  │   │
│                                                 └──────────────┬───────────────┘   │
└────────────────────────────────────────────────────────────────┼──────────────────┘
                                                                  ▼
                                             https://rkanri.genech.co.jp/kanri/nippo/*
```

### 3.1 プロセス構成とセキュリティ設定（main.js）

- `BrowserWindow` の `webPreferences`:
  - `contextIsolation: true` / `nodeIntegration: false`（レンダラーに Node を露出しない）

## 4. 中核処理：ページコンテキストでの API 実行

`renderer.js` の `callApi(endpoint, body, label)` が中核。手順は以下。

1. `<webview>`（id=`rk`）が `dom-ready` 済みか確認（未ログイン/未読込なら中断）。
2. `buildInjection(endpoint, body)` で、対象ページ内で実行する fetch コードを生成。
3. `rk.executeJavaScript(code)` で**対象ページのオリジン上**で実行。戻り値（Promise）を await。
4. 結果を結果テーブル / JSON 欄に表示。

### 4.1 注入する fetch（対象ページ内で実行される）

- URL: 相対パス `/kanri/nippo/<endpoint>`（same-origin）
- `method: POST` / `credentials: 'include'`
- ヘッダー:
  - `accept: application/json, text/plain, */*`
  - `content-type: application/json`
  - `x-xsrf-token`: **`document.cookie` の `XSRF-TOKEN` を URL デコードした値**
    - Laravel の `XSRF-TOKEN` Cookie は非 httpOnly のため JS から読める。
    - セッション Cookie は `credentials: include` により自動送出。
- `body`: 送信オブジェクトの JSON 文字列
- 戻り値: `{ ok, status, url, hasToken, data }`（`data` は JSON パース、失敗時は生テキスト）

### 4.2 ログイン切れの検出

- レスポンスが文字列で `<html` / `<!doctype` / `login` を含む場合、未ログインと判定し、
  「下の画面でログインしてください」と案内（接続ステータスも警告表示）。

### 4.3 接続ステータス表示

- webview の `did-start-loading` / `dom-ready` / `did-fail-load` を監視し、上部のドット（緑=OK / 橙=読込中 / 赤=失敗）とテキストで表示。
- `did-fail-load` の `errorCode === -3`（ABORTED、リダイレクト等）は無視。

- 「画面表示」ボタン（埋め込みブラウザの表示/非表示トグル）

### 5.2 操作パネル（左）

| ブロック | 項目 | 対応 API / 動作 |

| 日付/合計 | 日付(`date`) / 一昨昨日・一昨日・昨日・今日ボタン / 合計取得 / 日報一覧取得 | `getTotal`, `getNippoList` |
| 登録/更新 | 日報コード(`nippoCd`) / モードバッジ / コピー / クリア / 案件コード・取引先名・案件名(readonly) / 作業内容(select) / 作業時間 / 作業進捗(select) / 遅れ報告 / その他報告 / 登録・更新 / 削除 | `registData`, `deleteData` |

- 日付ボタンは現在日付 +offset を `yyyy/m/d`（ゼロ埋めなし）で設定（現行の `setDate` 準拠）。
- 起動時に日付へ当日をセット。

### 5.3 選択肢（現行仕様準拠）

- 作業内容(`sagyoNaiyo`): 空欄 + `1.管理` / `2.PJ会議` / `2.PJレビュー` / `3.調査・分析` / `4.要件定義` / `5..設計` / `6.開発・構築` / `7.テスト` / `8.ドキュメント作成` / `9.リリース対応` / `※作業待ち` / `※プレ・見積` / `※不具合` / `※問い合わせ` / `※社内会議` / `※レビュー支援` / `※学習・教育` / `※その他`
- 作業進捗(`sagyoShinchoku`): 空欄 / `順調` / `やや遅れ` / `遅れ`（既定 `順調`）

### 5.4 結果表示（右）

- 結果ラベル + ステータス（OK/NG・XSRF 取得有無）
- **結果テーブル**: 配列レスポンスを表示。行はクリック可能。
  - **表示列の絞り込み**: エンドポイント別の表示列ホワイトリスト（`DISPLAY_COLS`）を持ち、該当する列のみ表示する。`getNippoList` は実測で25フィールド返るため（[現行仕様.md](現行仕様.md) §7.4）、現行画面の表示列（日付 / 案件コード / 取引先名 / 案件名 / 作業時間 / 作業内容 / 作業進捗 / その他報告事項）に絞る。
  - ホワイトリスト未定義のエンドポイント、または該当列が1つも無い場合は、従来どおり先頭要素のキーから全列を自動生成する（フォールバック）。
  - **非配列応答は既存テーブルを維持**: `getTotal`（スカラー）や `registData`/`deleteData`（`{status}` オブジェクト）は一覧ではないため、テーブルを再描画せず直前の表示（日報一覧など）をそのまま残す。配列だが空（`[]`）の場合のみ「（一覧データなし）」を表示する。
  - **見出しラベル**: 生キーが英字等で分かりにくい場合、`HEADER_LABELS` で日本語見出しに置換する。現状 `getTantoList` の `key`→`担当者コード` / `名称1`→`担当者名`（[現行仕様.md](現行仕様.md) §6.2）。`getNippoList`/`gethistory` はキーが既に日本語のため置換不要。
  - **行データは全フィールド保持**し、表示列を絞っても行クリック時のフォーム反映（`日報コード`・`遅れ報告` 等の非表示列を含む）は影響を受けない。
- **生レスポンス(JSON)**: `<details>` で折りたたみ表示（全フィールドはここで確認可能）。
- **埋め込みブラウザ**: `<webview>` で対象ページを表示。初回ログイン導線を上部ヒントで案内。

### 5.5 行クリック時の反映（現行の setAnken / setTanto 相当）

| 実行 API | クリック行 | 反映内容 |
|---|---|---|
| `getNippoList` | 日報行 | 案件情報 + `nippoCd`・日付・作業内容・作業時間・作業進捗・遅れ報告・その他報告 をフォームへ → **編集モード**へ |

## 6. API 操作対応表

いずれも `callApi` 経由で `POST /kanri/nippo/<endpoint>` を実行。詳細は [現行仕様.md](現行仕様.md) §11。

| UI 操作 | endpoint | 送信 body |
|---|---|---|
| 日報一覧取得 | `getNippoList` | `{ tanto }` |
| 合計取得 | `getTotal` | `{ tanto, date }` |
| 登録 / 更新 | `registData` | 全11項目（`nippoCd` 空=新規 / 値あり=更新） |
| 削除 | `deleteData` | `{ nippoCd }` |
| 案件検索 | `gethistory` | `{ tanto, svalue }` |
| 案件履歴 | `gethistory` | `{ tanto }` |
| 担当者検索 | `getTantoList` | `{ svalue }` |

### 6.1 登録・更新の送信項目（全11）

`nippoCd, tanto, date, ankenCd, torihikisakiMei, ankenMei, sagyoNaiyo, sagyozikan, sagyoShinchoku, okureHokoku, sonotaHokoku`

### 6.2 業務ロジック（現行準拠）

- **モード判定**: `nippoCd` が空=新規モード、値あり=編集モード（バッジ表示）。
- **コピー**: `nippoCd` を空にするのみ（他項目は保持）→ 新規化。
- **登録/更新成功後**: フォーム初期化 → `getNippoList` で一覧再取得。
- **削除**: `nippoCd` 空なら中断。確認ダイアログ（「削除します。元に戻せませんのでご注意を」）承認後に実行 → 初期化 → 一覧再取得。

#### 成否判定（`status` ベース）

`registData` / `deleteData` は HTTP 200 でもボディ `{"status":"success"}` を返す（[現行仕様.md](現行仕様.md) §8.3/§8.4）。本アプリは現行画面でコメントアウトされていた `status` 判定を**実装として有効化**する。

- `isBizOk(r)` = HTTP OK **かつ**（応答が `{status:...}` 形式なら `status === "success"`）。
- `isBizOk` が真のときのみ「成功」toast → フォーム初期化 → 一覧再取得を実行。
- HTTP 200 でも `status !== "success"` の場合は**失敗扱い**（フォーム保持・一覧再取得なし）。
- **エラー時はサーバー応答をそのまま表示する**（canned な日本語メッセージに変換しない）。エラー時の応答形（`status` 値や構造）が未確認のため、`rawText()` で応答本体（オブジェクトは JSON 文字列化／文字列はそのまま）をステータス欄と toast に出力する。生レスポンスは従来どおり `<details>` にも表示。
- 一覧系（`getNippoList`/`gethistory`/`getTantoList`＝配列）・`getTotal`（スカラー）は `{status}` を持たないため、従来どおり HTTP OK で成功扱い。HTTP エラー（非 2xx）時も応答本体をそのまま表示。

### 6.3 入力バリデーション（`checkError` 相当）

上から評価し、最初に該当したものを toast 表示して中断。

1. `tanto` 空 → 担当者が未入力です
2. `date` 空 → 日付が未入力です
3. `date` 不正（`new Date` が NaN） → 日付が不正です
4. `sagyoNaiyo` 空 → 作業内容が未入力です
5. `sagyozikan` 空 → 作業時間が未入力です
6. `sagyozikan` 非数値 → 作業時間が不正です
7. `sagyoShinchoku` 空 → 作業進捗が未入力です

## 7. ビルド・実行

| コマンド | 内容 | 状態 |
|---|---|---|
| `npm install` | 依存（electron / electron-packager / electron-builder）を導入 | ✅ 確認済み |
| `npm start` | 開発起動（`electron .`） | ✅ 起動確認済み |
| `npm run pack` | electron-packager で実行可能な exe（フォルダ同梱形式）を生成 | ✅ 生成・起動確認済み |
| `npm run dist` | electron-builder で単一 exe（portable）を生成 | ⚠ 要 Developer Mode（下記） |
| `npm run dist:nsis` | electron-builder でインストーラ（NSIS）生成 | ⚠ 要 Developer Mode |

- **推奨（検証済み）**: `npm run pack` → `dist/NippoHokoku-win32-x64/NippoHokoku.exe`（約 270MB のフォルダごと配布。実行のみで起動、インストール不要）。
- WebView2 相当は Electron 同梱の Chromium を使用するため、別途ランタイム導入は不要。

### 7.1 単一 exe（electron-builder）に関する既知の制約

- `npm run dist` は署名ツール（winCodeSign）展開時に **Windows のシンボリックリンク作成権限**を要求する（macOS 用 `.dylib` シンボリックリンクを含むため）。
- 権限がない環境では `7za` が「必要な特権を保有していません」で失敗する。
- 回避策: **開発者モード（Developer Mode）を ON**、または管理者権限ターミナルで実行。不要なら `npm run pack` を使う。

## 8. セキュリティ・運用上の留意点

- **認証は本人のブラウザセッションに依存**。アプリは本人の権限・データ範囲でのみ動作する。認証情報をアプリが保存・送信することはない（Cookie は webview の `persist:rkanri` パーティションが保持）。
- **XSRF トークンはハードコードしない**。実行時に Cookie から都度取得する。
- 社内ネットワーク（LAN/VPN）接続時のみ対象ページに到達できる前提。オフライン/社外では webview 読込が失敗し、接続ステータスが赤表示になる。
- `registData` はサーバー応答の `status` によるエラー判定が現行画面ではコメントアウトされている（[現行仕様.md](現行仕様.md) §8.3）。本アプリでは HTTP ステータスと本文で成否を判定しているが、業務エラー（重複・制限超過等）の詳細レスポンス仕様は要確認。
