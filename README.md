# 日報登録プロジェクト

## 概要

社内LAN、または社内VPNに接続した端末からしかアクセスできないWebページ上で動作する **fetcher** を利用し、その取得・送信結果をもとに **API を実行する処理** を開発することを目的とするプロジェクトです。

対象は社内の日報登録画面（`https://rkanri.genech.co.jp/kanri/nippo`）で、この画面が内部で呼び出している API を、fetcher 経由で実行することを想定しています。

最終的な成果物としては、**exe ファイルを実行して起動し、自前の画面上から対象 API を実行できるデスクトップアプリ** として開発したいと考えています。

> 実装可否は現時点では未確定です。社内ネットワーク環境・セキュリティ要件・API 仕様に応じて構成を調整する前提で検討します。

---

## 背景・課題

日報登録画面は Vue 3 + axios で構成されており、各種操作は `/kanri/nippo` 配下の API を `POST`（`Content-Type: application/json`、`credentials: include`、XSRF トークン付与）で呼び出しています。

この画面は **社内LAN / 社内VPN 接続端末からのみアクセス可能** なため、外部からの自動化ができません。そこで、認証済みブラウザ上で動作する fetcher を使い、画面操作を介さずに日報登録・更新・削除・一覧取得などの業務処理を実行できるようにしたい、という要望があります。

---

## 目的

- 社内限定環境（LAN / VPN）で動作する fetcher を実装する
- fetcher で取得したデータや入力値をもとに、対象 API を呼び出す処理を作る
- 日報登録に関わる社内業務の自動化・効率化を進める

---

## 想定する構成

前提として、fetcher は **社内ネットワークに接続済みで、対象画面にログイン済みのブラウザ上** で動作します。既存セッション（Cookie / XSRF トークン）をそのまま利用することで、認証を別途実装せずに same-origin で API を呼び出す構成を想定しています。

```
[社内LAN / VPN 接続端末]
        │
        ▼
[ログイン済みブラウザ上の fetcher]
        │  同一オリジン(/kanri/nippo)へ POST
        │  credentials: include / XSRF トークン付与
        ▼
[日報登録 API 群]  ── 処理結果 ──▶ [自動処理・後続ロジック]
```

処理の流れ:

1. 社内ネットワーク内の対象 Web ページへアクセスし、ログイン済みセッションを確立する
2. fetcher で必要な情報（担当者・案件・既存日報など）を取得する
3. 取得結果を API リクエストの入力として組み立てる
4. 対象 API を実行し、処理結果を受け取って後続処理に利用する

---

## 希望する実装形態（デスクトップアプリ）

本プロジェクトで実現したいのは、**exe を実行して起動するデスクトップアプリ**です。アプリ内に用意した画面から、対象の日報登録 API（一覧取得・登録・更新・削除・合計取得・マスタ検索）を実行できることを目指します。

### 要望の要点

- **配布・起動**: exe ファイル単体で配布・実行できる（インストーラ有無は検討）。社内端末で起動するだけで使える状態を想定。
- **操作画面**: アプリ独自の GUI 画面を持ち、入力・ボタン操作で API を実行できる。既存 Web 画面を開かなくても日報の登録・更新・削除・一覧確認ができることが理想。
- **API 実行**: [現行仕様.md](現行仕様.md) に整理した各 API を、アプリから直接呼び出す。
- **社内ネットワーク前提**: 社内 LAN / VPN 接続端末からのみ動作すればよい（外部公開は不要）。

### 希望する画面フロー（UX）

操作画面は、以下のフローで動作させたい。登録・マスタ検索は**ポップアップ（モーダル）表示**とし、メイン画面は日報一覧を中心とするシンプルな構成にする。

1. **認証画面**を表示する（対象システム＝Microsoft SSO にログイン）。
2. 認証後、以下を表示する。
   - 日報一覧テーブル
   - 登録ボタン
3. 一覧の明細（行）を押下したとき、**修正モード**で登録画面（登録エリアのみ）を**ポップアップ表示**する。
4. 登録ボタンを押下したとき、**新規モード**で登録画面（登録エリアのみ）を**ポップアップ表示**する。
5. 登録画面の**案件コードの右に検索ボタン**を追加する。案件の検索ボタン押下時に**案件マスタ一覧をポップアップ表示**し、一覧明細を押下したら**案件コード・取引先名・案件名**を登録画面に設定する。
6. 登録画面の**担当の検索ボタン**押下時に**担当者マスタ一覧をポップアップ表示**し、一覧明細を押下したら**担当者コード・担当者名**を登録画面に設定する。

> 画面の詳細（モーダル構成・項目・イベント）は [新仕様.md](新仕様.md) §5 を参照。

### 想定するアプリ構成（案）

デスクトップアプリからでも、API は `credentials: include` と **セッション毎に変わる XSRF トークン** を要求するため、認証の取り回しが実装上の要になります。現実的な構成案は以下のとおりです。

- **案A: 組み込みブラウザ（WebView）方式**
  - アプリ内に WebView を持ち、対象ページにログインしてもらう（または既存セッションを利用）。
  - WebView のコンテキスト（＝ログイン済みセッション・XSRF トークンが有効な状態）から fetch を実行する。
  - Cookie / XSRF の取り回しをブラウザ側に委ねられるため、認証実装が最も軽い。
  - 技術候補: Electron / Tauri / .NET(WebView2) など。

- **案B: ネイティブ HTTP クライアント方式**
  - アプリが直接 HTTP リクエストを送る。
  - ログイン処理を行ってセッション Cookie を取得し、`XSRF-TOKEN` Cookie を復号して `x-xsrf-token` ヘッダーに付与する必要がある。
  - 認証・トークン処理を自前で実装する必要があり、案Aより難度が高い。

> 現時点では **案A（組み込みブラウザ方式）** が、既存の認証機構をそのまま活かせるため有力と考えられます。技術スタックは今後選定します。

## 対象システム仕様（要点）

詳細は [現行仕様.md](現行仕様.md) を参照してください。ここでは fetcher / API 実行に必要な要点のみ抜粋します。

### 対象画面

- URL: `https://rkanri.genech.co.jp/kanri/nippo`
- 構成: Vue 3 + axios
- 画面は「登録エリア」「マスタ一覧エリア」「日報一覧エリア」の3ブロック
- 実入力対象は主に、日付・作業内容・作業時間・作業進捗・自由記述欄。案件／担当者情報はマスタ一覧の選択により自動反映される

### API 共通仕様

- メソッド: `POST`
- `Content-Type: application/json`
- `credentials: include`（既存セッションを利用）
- XSRF トークンをヘッダーに付与
- 送受信は same-origin の `/kanri/nippo` 配下
- ボディは JSON 文字列

### 主なエンドポイント

| 用途 | エンドポイント | 主な送信データ |
|---|---|---|
| 日報一覧取得 | `/kanri/nippo/getNippoList` | `tanto` |
| 新規登録 / 更新 | `/kanri/nippo/registData` | `nippoCd`（空=新規 / 値あり=更新）ほかフォーム全項目 |
| 削除 | `/kanri/nippo/deleteData` | `nippoCd` |
| 合計作業時間取得 | `/kanri/nippo/getTotal` | `tanto`, `date` |
| 案件一覧 / 履歴取得 | `/kanri/nippo/gethistory` | `tanto`, `svalue`(任意) |
| 担当者一覧取得 | `/kanri/nippo/getTantoList` | `svalue`(任意) |

### 登録・更新の送信項目

`nippoCd`, `tanto`, `date`, `ankenCd`, `torihikisakiMei`, `ankenMei`, `sagyoNaiyo`, `sagyozikan`, `sagyoShinchoku`, `okureHokoku`, `sonotaHokoku`

- `nippoCd` が空文字なら新規登録、既存コードを持つ場合は更新として扱われる
- `date` は `yyyy/m/d` 形式
- `sagyozikan`（作業時間）は数値文字列
- `sagyoShinchoku`（作業進捗）の既定値は「順調」

登録リクエスト例:

```json
{
  "nippoCd": "",
  "tanto": "P-000184",
  "date": "2026/7/7",
  "ankenCd": "A-001874",
  "torihikisakiMei": "株式会社スーパー・コート",
  "ankenMei": "新介護記録システム構築支援費用（結合テスト）",
  "sagyoNaiyo": "6.開発・構築",
  "sagyozikan": "8",
  "sagyoShinchoku": "順調",
  "okureHokoku": "",
  "sonotaHokoku": "結合テスト仕様書レビュー"
}
```

### 入力バリデーション（現行画面基準）

- 担当者コード未入力: 「担当者が未入力です」
- 日付未入力 / 形式不正: 「日付が未入力です」「日付が不正です」
- 作業内容未入力: 「作業内容が未入力です」
- 作業時間未入力 / 非数値: 「作業時間が未入力です」「作業時間が不正です」
- 作業進捗未入力: 「作業進捗が未入力です」

## 実装状況

案A（WebView 組み込み / Electron）で実装を進行中。アプリの設計・実装仕様は [新仕様.md](新仕様.md) にまとめています。

- 認証方式は **Microsoft（Azure AD / Entra ID）SSO** であることを確認済み（webview 内でログイン）。
- 全 API（一覧取得・登録/更新・削除・合計取得・案件/担当マスタ検索）を自前 UI から実行可能。

### 開発・ビルド手順

前提: Node.js（v24 で確認）/ npm。社内 LAN・VPN 接続端末。

```sh
npm install        # 依存（electron / electron-packager / electron-builder）を導入
npm start          # 開発起動（アプリを立ち上げて動作確認）
npm run pack       # 実行可能な exe を生成（推奨・検証済み）
```

- `npm run pack` の生成物: `dist/NippoHokoku-win32-x64/`（フォルダ）。この中の **`NippoHokoku.exe` を実行するとアプリが起動**します（インストール不要）。配布はこのフォルダごと行います（約 270MB）。
- 初回はアプリ内の埋め込み画面で対象システム（Microsoft SSO）にログインしてください。ログイン後、左パネルの操作から各 API を実行できます。

#### 単一 exe（portable）で配布したい場合

`npm run dist`（electron-builder）で単一 exe を生成できますが、本環境では **Windows のシンボリックリンク作成権限**が必要です（署名ツール展開のため）。以下のいずれかを実施してから実行してください。

- Windows の **開発者モード（Developer Mode）を ON**（設定 > プライバシーとセキュリティ > 開発者向け）、または
- 管理者権限のターミナルで実行

未対応の場合は上記 `npm run pack`（フォルダ + exe）で配布してください。

> 補足: 開発 PC の環境変数に `ELECTRON_RUN_AS_NODE=1` が設定されていると、`electron` が GUI ではなく Node として起動します。GUI 起動時はこの変数を解除してください。
