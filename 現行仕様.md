# 現行仕様

共有された HTML（新規モード / 編集モード）と Vue スクリプトを解析し、実装に直結する情報（画面構成・データモデル・入力状態・選択肢・キー名・API 送受信・制御ロジック）を整理した仕様書である。HTML 本体そのものは記載せず、要点のみ抜粋する。

---

## 1. 対象ページ

- URL: `https://rkanri.genech.co.jp/kanri/nippo`
- 実装: Vue 3（`Vue.createApp(...).mount("#app")`）+ axios
- 通信: axios.post による same-origin `/kanri/nippo/*` への POST（ブラウザ上は fetch 相当の POST）

---

## 2. 画面構成

画面は **入力エリア（上部）** と **日報一覧エリア（下部）** の2ブロック構成。
入力エリアはさらに **登録・更新エリア（左部）** と **マスタ一覧エリア（右部）** に分かれる。

全体を大きく3ブロックとして、以降は次の名称で呼ぶ。

1. **① 登録エリア**（入力エリア左部 / `float:left`）
2. **② マスタ一覧エリア**（入力エリア右部 / `float:left; margin-left:20px`）
3. **③ 日報一覧エリア**（画面下部 / `<hr>` 以下）

```
┌──────────────────────────────────────────────┐
│ 入力エリア（上部）                              │
│ ┌───────────────────┐ ┌────────────────────┐ │
│ │ ① 登録エリア        │ │ ② マスタ一覧エリア   │ │
│ └───────────────────┘ └────────────────────┘ │
├──────────────────────────────────────────────┤
│ ③ 日報一覧エリア（下部）                        │
└──────────────────────────────────────────────┘
```

### 4.2 モードによる日報コード表示の切替

- **新規モード**: 日報コードのセルに `<span>新規登録</span>` を表示（`nippoCd` が空）。コピーボタンは非表示。
- **編集モード**: `nippoCd` を readonly input（グレー背景）で表示し、**コピーボタン**を表示。
- 切替は `nippoCd` の有無で制御される（`v-if` 相当）。

### 4.3 編集モードへの遷移

- ③ 日報一覧の行をクリックすると `setAnken(行データ, 行データの日報コード)` が呼ばれ、フォーム各項目に値が反映されて編集モードになる（§7.3 参照）。
- 別途「編集専用画面」への遷移はなく、同一画面内でフォームが書き換わる。

## 6. ② マスタ一覧エリア

`history` により表示が切り替わる。初期は `history=true`（案件検索モード）。

### 6.1 案件検索モード（`history=true`、初期表示のデフォルト）

構成:
- 検索テキスト（`svalue`、案件コード・取引先名・案件名の部分一致想定）
- 検索ボタン → `getHistory(1)`
- 履歴ボタン → `getHistory(0)`
- 案件一覧テーブル（`sdata` を表示）
  - 表示列: **案件コード / 取引先名 / 案件名**（`sdata` 各要素のキー: `案件コード`, `取引先名`, `案件名`）
  - 行クリックで `setAnken(sd)` → 登録エリアの案件情報に反映

> `sdata` は初期表示では空（`mounted` では取得しない）。検索ボタンまたは履歴ボタン押下で取得される。

### 6.2 担当者検索モード（`history=false`、登録エリアの検索ボタン押下時に表示）

構成:
- 検索テキスト（`svalue2`、担当者コード・担当者名の部分一致想定）
- 検索ボタン → `searchTanto(1)` 相当（`getTantoList`）
- キャンセルボタン → `searchTanto(0)`（`history=true` に戻す）
- 担当者一覧テーブル（`tantoList` を表示）
  - 表示列: **担当者コード / 担当者名**
  - **`tantoList` 各要素のキー: `key`（担当者コード）, `名称1`（担当者名）**
  - 行クリックで `setTanto(item)` → `tanto=item.key`, `tantoName=item.名称1` を設定し `history=true` に戻す

## 7. ③ 日報一覧エリア

### 7.1 表示

- `<hr>` 下のテーブルに `nippoList` を表示。各行 `style="cursor: pointer;"`（クリック可能）。
- **表示列（実 HTML 準拠）**: 日付 / **案件コード** / 取引先名 / 案件名 / 作業時間 / 作業内容 / 作業進捗 / その他報告事項

> **修正点**: 従来「2列目=日報コード」と記載していたが、実 HTML の表示は **案件コード**（`A-xxxxxx`）である。日報コード（`G-xxxxxx`）は表示列にはなく、行データ内部に保持され、クリック時に編集対象の特定に使われる。

### 7.2 行データ（`nippoList` 要素）のキー

行クリック時に参照されるため、各要素は少なくとも以下の日本語キーを持つ。

- `日付`, `案件コード`, `取引先名`, `案件名`, `作業時間`, `作業内容`, `作業進捗`, `遅れ報告`, `その他報告事項`, `日報コード`

（`遅れ報告` と `日報コード` は一覧に表示されないが、行クリック時のフォーム反映に使用される。）

### 7.3 行クリック時の挙動（編集モード遷移）

`setAnken(sd, nippo)` が呼ばれ、以下を反映する。

- 常時: `ankenCd=sd.案件コード`, `torihikisakiMei=sd.取引先名`, `ankenMei=sd.案件名`
- `nippo`（＝日報コード）が空でない場合のみ追加で:
  - `nippoCd=nippo`
  - `date=sd.日付`, `sagyoNaiyo=sd.作業内容`, `sagyozikan=sd.作業時間`, `sagyoShinchoku=sd.作業進捗`, `okureHokoku=sd.遅れ報告`, `sonotaHokoku=sd.その他報告事項`

→ 日報一覧の行クリックは `setAnken(row, row.日報コード)`、案件一覧（②）の行クリックは `setAnken(sd)`（第2引数なし＝案件情報のみ反映）。

## 8. メソッド／制御ロジック詳細

### 8.1 `setDate(p)` — 日付ボタン

- 現在日付に `p` 日を加算し、`date` に `年/月/日` 形式で設定（月・日はゼロ埋めなし ＝ `yyyy/m/d`）。
- 割当: 今日=`0`、昨日=`-1`、一昨日=`-2`、一昨昨日=`-3`。

### 8.2 `copyData()` — コピー

- `nippoCd = ""` にするのみ。他の入力値はすべて維持したまま新規モード相当になる。

### 8.3 `registData()` — 日報登録／更新

- 先に `checkError()` を実行し、`false` なら中断。
- 送信キー: `nippoCd, tanto, date, ankenCd, torihikisakiMei, ankenMei, sagyoNaiyo, sagyozikan, sagyoShinchoku, okureHokoku, sonotaHokoku`
- `nippoCd` が空＝新規、値あり＝更新（エンドポイントは共通 `/registData`）。
- 成功後: `nippoCd`・入力各項目を初期化し、`getNippoList()` で一覧を再取得。
- コード上、レスポンス `status` を見て「API制限値を超えています。１〜２分時間をおいてから実行してください」と警告する処理が**コメントアウトで存在**（現状は無効）。

### 8.4 `deleteData()` — 削除

- `nippoCd == ""` なら何もしない。
- `confirm("削除します。元に戻せませんのご注意を")` で承認時のみ実行。
- 送信キー: `nippoCd` のみ。
- レスポンスの `status` を `alert` 表示。入力初期化＆一覧再取得。

### 8.5 `getHistory(m)` — 案件一覧／履歴取得

- 常に `tanto` を送信。`m != 0` の場合のみ `svalue` を追加。
- 検索ボタン=`getHistory(1)`（`{tanto, svalue}`）、履歴ボタン=`getHistory(0)`（`{tanto}`）。
- 結果を `sdata` に格納。

### 8.6 `searchTanto(mode)` — 担当者検索モード切替／検索

- `mode == 0`: `history=true` に戻して終了（＝キャンセル）。
- `mode != 0`: `history=false`、`{svalue: svalue2}` を `getTantoList` に送信し `tantoList` に格納。

### 8.7 `setTanto(item)` — 担当者選択

- `tanto=item.key`, `tantoName=item.名称1`, `history=true`。
- `getNippoList()` を実行し、日付欄へフォーカス。

### 8.8 `getNippoList()` — 日報一覧取得

- 送信キー: `{tanto}`（引数は無視し `this.tanto` を使用）。
- 結果を `nippoList` に格納。

### 8.9 `getTotal()` — 合計時間取得

- `date` が正規表現 `/\d{4}\/\d{1,2}\/\d{1,2}$/` に一致し、かつ有効な日付の場合のみ実行。
- 送信キー: `{tanto, date}`。結果を `total`（合計時間ラベル）に反映。

### 8.10 `checkError()` — 入力チェック（§9）

## 9. 入力チェック・バリデーション（`checkError()`）

上から順に評価し、最初に該当したものを `alert` してから中断する。

| 条件 | メッセージ |

## 11. API 送受信

### 11.1 共通リクエスト仕様

- メソッド: `POST`（axios.post）
- `Content-Type: application/json`
- same-origin `/kanri/nippo/*` 配下
- `credentials: include`（ブラウザの既存セッション利用）／XSRF トークン付与を前提
- ボディは JSON 文字列

### 11.2 エンドポイント一覧

| メソッド | エンドポイント | 送信データ | レスポンス格納先 |
|---|---|---|---|
| `getNippoList` | `/kanri/nippo/getNippoList` | `{tanto}` | `nippoList`（§7.2 のキー） |
| `registData` | `/kanri/nippo/registData` | フォーム全11項目（§8.3） | ―（成功後に一覧再取得。`status` 参照はコメントアウト） |
| `deleteData` | `/kanri/nippo/deleteData` | `{nippoCd}` | `status` を alert |

### 11.4 実 fetch リクエスト仕様（共通）

ブラウザ実通信は以下の形の `fetch` として送信される（DevTools の Copy as fetch 相当）。

- `method`: `"POST"`
- `mode`: `"cors"`
- `credentials`: `"include"`（Cookie セッションを送出）
- `referrer`: `"https://rkanri.genech.co.jp/kanri/nippo"`
- `body`: JSON 文字列（エンドポイント別は §11.5）

共通ヘッダー:

| ヘッダー | 値 |
|---|---|
| `accept` | `application/json, text/plain, */*` |
| `accept-language` | `ja,en-US;q=0.9,en;q=0.8` |
| `content-type` | `application/json` |
| `priority` | `u=1, i` |
| `sec-ch-ua` | `""` |
| `sec-ch-ua-mobile` | `?0` |
| `sec-ch-ua-platform` | `""` |
| `sec-fetch-dest` | `empty` |
| `sec-fetch-mode` | `cors` |
| `sec-fetch-site` | `same-origin` |
| `x-xsrf-token` | XSRF トークン（下記）※リクエスト毎に異なる |

> **重要（XSRF トークン）**: `x-xsrf-token` は Laravel 系の暗号化トークンで、**セッション毎・（実測では）リクエスト毎に値が変わる**。ハードコード不可。fetcher 実装時は、対象ページのコンテキストで Cookie `XSRF-TOKEN` を復号して付与するか（axios は既存 Cookie から自動付与している）、ログイン済みブラウザ上で実行して既存の付与機構に委ねる必要がある。共有された各トークン文字列はサンプルであり、他セッションでは無効。

### 11.5 エンドポイント別 fetch（`body` 抜粋）

いずれも上記共通ヘッダー・共通オプションを用い、`body` のみ異なる。

| # | 操作 | URL | `body`（JSON） |
|---|---|---|---|
| ① | 日報一覧取得 | `/kanri/nippo/getNippoList` | `{"tanto":"P-000184"}` |
| ② | 新規登録 | `/kanri/nippo/registData` | `{"nippoCd":"", ... }`（全11項目、`nippoCd` 空） |
| ③ | 更新 | `/kanri/nippo/registData` | `{"nippoCd":"G-045674", ... }`（全11項目、`nippoCd` 値あり） |
| ④ | 削除 | `/kanri/nippo/deleteData` | `{"nippoCd":"G-045674"}` |
| ⑤ | 合計取得 | `/kanri/nippo/getTotal` | `{"tanto":"P-000184","date":"2026/7/7"}` |
| ⑥ | 案件一覧（検索） | `/kanri/nippo/gethistory` | `{"tanto":"P-000184","svalue":""}` |
| ⑦ | 案件一覧（履歴） | `/kanri/nippo/gethistory` | `{"tanto":"P-000184"}` |
| ⑧ | 担当一覧（検索） | `/kanri/nippo/getTantoList` | `{"svalue":""}` |

補足:
- 新規登録（②）と更新（③）は同一 URL `/registData`。`nippoCd` が空か否かでサーバー側が新規/更新を判定する（§8.3 と整合）。
- 案件一覧は同一 URL `/gethistory` で、`svalue` キーの有無により「検索（⑥、`svalue` あり）」と「履歴（⑦、`svalue` なし）」を切り替える（§8.5 の `getHistory(m)` と整合）。
- ②③ の全11項目の `body` 例は §11.3 を参照。

## 12. 補足・留意点

- 入力欄の多くは readonly。実際に手入力するのは **日付・作業内容・作業時間・作業進捗・遅れ報告・その他報告事項** の6項目のみ。案件情報・担当者情報はマスタ一覧の選択で自動設定される。
- 案件コード（`A-xxxxxx`）と日報コード（`G-xxxxxx`）は別物。③ 一覧の2列目に見えるのは案件コードで、日報コードは内部データとして保持される。
- `registData` はレスポンス `status` によるエラーハンドリングが現状コメントアウトされているため、実運用では成功前提で一覧が初期化される点に注意。
- `getNippoList` は引数を無視し常に `this.tanto` を送るため、担当者切替（`setTanto`）後は最新 `tanto` で再取得される。
- **fetcher 実装上の最重要点**: 全 API が `credentials: include` + `x-xsrf-token`（セッション毎に異なる）に依存する。外部プロセスからの直接呼び出しは XSRF トークン・Cookie の取り回しが必須のため、**ログイン済みブラウザのコンテキスト内で fetcher を動かす構成**（ブックマークレット／拡張機能／画面内スクリプト等）が現実的である（§11.4 参照）。
