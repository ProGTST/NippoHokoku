# リリース手順（新バージョンの配信）

日報入力アプリの新バージョンを配信し、利用者が自動更新できるようにするための手順書。
仕組みの詳細は [新仕様.md](新仕様.md) §7.2「自動更新」を参照。

- **配信元**: GitHub Releases（`ProGTST/NippoHokoku`。**private リポジトリ**）
- **更新ライブラリ**: electron-updater
- **配布形式**: NSIS インストーラ（自動更新は NSIS が前提）
- **private 対応**: 読み取り専用トークン（`UPDATE_TOKEN`）をビルド時にクライアントへ埋め込み、更新チェックに使う（[新仕様.md](新仕様.md) §7.2.1）。発行時に **2 種類のトークン**を使う点に注意（下記）。

---

## 0. 前提（初回のみ確認）

- [ ] **リポジトリは private 運用**: private でも自動更新できるよう、読み取り専用トークン（`UPDATE_TOKEN`）をビルド時にクライアントへ埋め込む方式で対応済み（§7.2.1）。
- [ ] **利用者端末から GitHub へ到達可能**（`github.com` / `objects.githubusercontent.com` / `api.github.com`）。到達不可なら generic provider 方式に変更。
- [ ] **利用者は NSIS 版をインストール済み**（`日報入力-Setup-x.x.x.exe`）。従来の portable exe からは自動更新できない。
- [ ] **発行用トークン `GH_TOKEN`（`repo` / Contents: Read and write 権限）** を取得済み（Release のアップロード用。§2）。
- [ ] **クライアント埋め込み用トークン `UPDATE_TOKEN`（Contents: Read のみの最小権限）** を取得済み（配布 exe に埋め込む更新チェック用。§2.2）。

---

## 1. バージョンを上げる

[package.json](package.json) の `version` を上げる（例 `0.1.0` → `0.1.1`）。

```json
{
  "version": "0.1.1"
}
```

> ⚠ electron-updater は semver で「リモート > 現在」のときのみ更新ありと判定する。**必ず上げること**。
> バージョン番号は [セマンティック バージョニング](https://semver.org/lang/ja/) 準拠（メジャー.マイナー.パッチ）。

---

## 2. GitHub トークンを設定

**2 種類のトークン**を設定する。役割が異なるので混同しない。

| 環境変数       | 役割                                                        | 権限                    | 埋め込み                          |
| -------------- | ----------------------------------------------------------- | ----------------------- | --------------------------------- |
| `GH_TOKEN`     | Release をアップロードする（発行用）                        | `repo` / Contents: R&W  | されない（発行 PC のみ）          |
| `UPDATE_TOKEN` | クライアントが private の Releases を読む（更新チェック用） | **Contents: Read のみ** | **配布 exe に埋め込む**（§7.2.1） |

PowerShell（同じセッションで両方設定）:

```powershell
$env:GH_TOKEN = "ghp_xxxxxxxxxxxxxxxxxxxx"      # 発行用
$env:UPDATE_TOKEN = "ghp_yyyyyyyyyyyyyyyyyyyy"  # クライアント埋め込み用（読み取り専用）
```

> - トークンはコミット・共有しないこと。セッション（ターミナル）ごとに設定する。
> - `UPDATE_TOKEN` は配布 exe に平文で入るため、**必ず Contents: Read のみの最小権限**にする（書き込み権限を付けない）。
> - `UPDATE_TOKEN` 未設定でも `npm run publish` は通るが、**その配布物は private の自動更新が失敗する**（`gen-token` が警告を出す）。private 運用では必ず設定すること。

### 2.1 トークンの取得方法

GitHub の Personal Access Token（PAT）を発行する。**Classic（従来型）** が確実。

> ⚠ **場所に注意**: トークンは**アカウント設定**の中の「Developer settings」で作る。**リポジトリの Settings タブではない**（リポジトリ設定には Developer settings は無い）。
> GitHub には Settings が 2 種類あり紛らわしい:
>
> - リポジトリの **Settings タブ**（Collaborators / Environments など）← ここには**無い**
> - **アカウントの Settings**（画面右上の丸いプロフィール画像 → Settings）← Developer settings は**こちら**

**Classic トークン（推奨）**

1. **一番簡単**: ブラウザで直接 **`https://github.com/settings/tokens`** を開く（リポジトリ画面ではなく個人アカウントのトークン画面へ飛ぶ）。
   - 手動で辿る場合: 画面**右上の丸いプロフィール画像**をクリック → **Settings** → 左メニュー最下部の **Developer settings** → **Personal access tokens → Tokens (classic)**。
2. **Generate new token → Generate new token (classic)**。
3. 設定:
   - **Note**: 用途がわかる名前（例 `nippo-release`）。
   - **Expiration**: 有効期限（例 90 日。無期限は非推奨）。
   - **Select scopes**: **`repo`** にチェック（private リポジトリの場合は必須。public でも `repo` を付けておけば確実）。
4. **Generate token** を押し、表示された **`ghp_...` をコピー**。
   > ⚠ トークンは**この画面でしか表示されない**。閉じると二度と見られないので必ず控える。
5. 上記の `$env:GH_TOKEN = "..."` に貼り付けて設定。

**Fine-grained トークン（代替）**

- 直リンク **`https://github.com/settings/personal-access-tokens/new`**（または アカウント Settings → Developer settings → **Fine-grained tokens**）から発行。
- **Repository access**: `ProGTST/NippoHokoku` を選択。
- **Permissions → Repository permissions → Contents: Read and write**（Release の作成・アップロードに必要）。
- 生成後の `github_pat_...` を `GH_TOKEN` に設定。

> - リポジトリが **他人/Organization（`ProGTST`）所有**の場合、そのリポジトリへの書き込み権限（コラボレーター/メンバー権限）が必要。
> - 有効期限が切れたら再発行して `GH_TOKEN` を更新する。
> - 毎回設定するのが手間なら、OS のユーザー環境変数に登録しておく（`setx GH_TOKEN "ghp_..."` → 新しいターミナルから有効。ただし端末に平文保存される点に注意）。

### 2.2 クライアント埋め込み用トークン（`UPDATE_TOKEN`）の取得

配布 exe に埋め込み、利用者端末が **private リポジトリの Releases を読む**ための**読み取り専用**トークン。exe に平文で入るため、**Contents: Read だけ**の最小権限で発行する（万一漏れても書き込み・他リポジトリ操作をさせない）。**Fine-grained トークン**推奨。

1. **`https://github.com/settings/personal-access-tokens/new`** を開く（アカウント Settings → Developer settings → **Fine-grained tokens**）。
2. 設定:
   - **Token name**: 用途がわかる名前（例 `nippo-update-readonly`）。
   - **Expiration**: 有効期限（切れると利用者の更新チェックが失敗するため、運用に合わせて長め＋失効前に更新）。
   - **Repository access**: **Only select repositories** → `ProGTST/NippoHokoku`。
   - **Permissions → Repository permissions → Contents: Read-only**（他の権限は付けない）。
3. **Generate token** で発行し、`github_pat_...` をコピー → `$env:UPDATE_TOKEN` に設定。

> - Classic トークンでも代用可だが `repo` スコープは private 全体への読み書きになり過剰。**Fine-grained + Contents: Read-only** を推奨。
> - このトークンはビルド時に `src/update-token.js`（`.gitignore` 済み）へ書き出され exe に同梱される。**git にはコミットされない**（仕組みは §7.2.1）。
> - 失効・更新したら**再ビルドして配布し直す**必要がある（既存 exe には旧トークンが埋め込まれているため）。

## 3. ビルド＆発行

```powershell
npm run publish
```

`npm run publish` = `npm run gen-token && electron-builder --win nsis --publish always`。以下を自動実行:

- **`gen-token`**: 環境変数 `UPDATE_TOKEN` から `src/update-token.js`（自動更新トークン）を生成（未設定なら警告して空トークン。§2）
- NSIS インストーラ `日報入力-Setup-<version>.exe` を生成（`dist/`）
- 更新判定用マニフェスト `latest.yml`・インストーラ・`.blockmap` を **GitHub Releases へアップロード**

> 実行ログに `[gen-token] src/update-token.js を生成しました（token: 有り）` と出ることを確認する。`token: なし` の場合は `UPDATE_TOKEN` 未設定＝**private の自動更新が効かない**ため、設定し直して再実行する。

生成物（`dist/`）の例:

| ファイル                            | 用途                                            |
| ----------------------------------- | ----------------------------------------------- |
| `日報入力-Setup-0.1.1.exe`          | NSIS インストーラ（新規配布・自動更新の実体）   |
| `latest.yml`                        | 更新判定マニフェスト（electron-updater が参照） |
| `日報入力-Setup-0.1.1.exe.blockmap` | 差分ダウンロード用                              |

---

## 4. Release を公開する

`npm run publish` は通常 **ドラフト（下書き）** で Release を作成する。

1. GitHub の **Releases** 画面を開く（`https://github.com/ProGTST/NippoHokoku/releases`）。
2. 該当バージョンのドラフトを開く。
3. タグ・タイトル・変更点を確認し、**「Publish release」** で公開。

> ⚠ **公開しないと利用者側の更新チェックに現れない**。

---

## 5. 動作確認

1. 旧バージョンの NSIS 版アプリを起動 → ログイン → 日報一覧を表示。
2. 確認ダイアログが出ることを確認:
   ```
   最新のバージョン 0.1.1 がアップロードされています。
   最新のバージョンにアップデートしますか？
   ```
3. **OK** → ダウンロード → 自動再起動 → フッター右下が `v0.1.1` になれば成功。

---

## 利用者側の挙動（参考）

- 起動しログイン → 日報一覧表示時に自動チェック。
- 更新があれば上記の確認ダイアログ。
  - **OK**: ダウンロード → 完了後に自動再起動して適用。
  - **キャンセル**: そのまま利用継続。フッター左下の **「⬆ 更新」ボタン** からいつでも更新可能。
- フッター右下に現在バージョン（`v<version>`）を常時表示。

## トラブルシューティング

| 症状                                    | 原因・対処                                                                                                                                                      |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 更新ダイアログが出ない                  | Release がドラフトのまま／`version` を上げ忘れ／端末から GitHub に到達不可／`UPDATE_TOKEN` 未設定でビルドした exe（private を読めない）                         |
| `GH_TOKEN` エラーで publish 失敗        | トークン未設定・権限不足（`repo` / Contents: R&W が必要）                                                                                                       |
| ダウンロード/更新確認で失敗（利用者側） | `UPDATE_TOKEN` 未設定・権限不足・失効した exe を配布している（private を読めない）／ネットワーク不通。`UPDATE_TOKEN` を設定して**再ビルド・再配布**する（§2.2） |
| `gen-token` が `token: なし` と出る     | `UPDATE_TOKEN` 未設定。§2 の手順で設定して再実行                                                                                                                |
| ビルドが winCodeSign で失敗             | [新仕様.md](新仕様.md) §7.1 の回避手順（キャッシュ手動展開 or 開発者モード）を参照                                                                              |
| portable 版利用者が更新されない         | portable は自動更新非対応。NSIS 版を配布・入れ直してもらう                                                                                                      |

```powershell
# 1. package.json の version を上げる（例 0.1.1）
# 2. トークン設定（発行用＋クライアント埋め込み用の2種）
$env:GH_TOKEN = "ghp_..."       # 発行用（repo / Contents: R&W）

# 4. GitHub Releases でドラフトを Publish
```
