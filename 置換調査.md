# 置換調査 — 本アプリをおおもとの Web サーバー上で運用できるか

対象システム（`https://rkanri.genech.co.jp/kanri/nippo`）のサーバー上に、本アプリ（日報入力）の画面を置いて Web で運用できないか、という相談についての技術調査と提案をまとめる。

関連: [新仕様.md](新仕様.md)（本アプリの設計）／[現行仕様.md](現行仕様.md)（対象システムの仕様）／[README.md](README.md)（背景）。

---

## 0. 前提整理：本アプリが「なぜ Electron／WebView なのか」

本アプリの画面（`src/index.html` / `src/styles.css` / `src/renderer.js`）は、**フレームワークを使わない素の HTML + CSS + JavaScript** で作られている。つまり画面そのものは Web 技術であり、本質的にはブラウザで動くものである。

にもかかわらず Electron + WebView という構成を採っているのは、**認証（same-origin 制約）を回避するため**である。

- 対象 API は `credentials: include`（Cookie セッション）と、セッション毎に変わる **XSRF トークン**を要求する。
- デスクトップアプリ（別オリジン）から直接 `fetch` しても、Cookie も XSRF トークンも送れないため API を実行できない。
- そこで **アプリ内に対象ページを読み込む `<webview>` を持ち、`webview.executeJavaScript()` で「対象ページのオリジン上」で fetch を実行**している（`renderer.js` の `buildInjection` / `callApi`）。これにより Cookie / XSRF が自動的に有効な状態で API を呼べる。

```
現状（Electron / 別オリジン）
┌─ Electron BrowserWindow ─────────────────────────────┐
│  自前UI(index.html)   executeJavaScript(fetch…)        │
│  ───────────────────▶ <webview src=…/kanri/nippo>     │
│  ◀─── 結果(JSON) ───── (ログイン済み・same-origin fetch)│
└───────────────────────────────────────────────────────┘
```

**ここが今回の相談の核心**：もし本アプリの画面を **rkanri と同じサーバー（同一オリジン）に置く**なら、画面自体がすでにログイン済み・same-origin になるため、**WebView も executeJavaScript も不要**になる。ページから直接 `fetch('/kanri/nippo/registData', {credentials:'include', headers:{'x-xsrf-token': …}})` を呼べばよく、これは現状 `buildInjection` が生成している fetch コードとほぼ同一である。

つまり **「サーバーに置く」＝「WebView 迂回をやめて、素の Web ページに戻す」** という方向であり、構成としてはむしろ現状より単純になる。

## ① ブラウザで表示できるか

**結論：画面レイヤーはそのままブラウザで表示できる。ただし現状のコードは一部を修正しないとブラウザ単体では動かない。**

### そのまま使える部分（大半）

- `index.html` / `styles.css` は純粋な HTML/CSS。ブラウザ（Chromium / Edge 等）でそのまま描画できる。
- `renderer.js` の業務ロジック（一覧表示・絞り込み・登録/更新/削除・一括コピー/変更/置換・モーダル・バリデーション等）は素の DOM 操作のみで、ブラウザでそのまま動く。
- 自動更新まわり（`window.appApi.*`）は既に **`if (window.appApi …)` でガード**されているため、ブラウザでは自然に無効化され、エラーにならない。

### 修正が必要な部分（Electron / WebView 依存）

| 箇所                                                                      | 現状                               | ブラウザ化での対応                                                               |
| ------------------------------------------------------------------------- | ---------------------------------- | -------------------------------------------------------------------------------- |
| `<webview id="rk">` と各種イベント（`dom-ready` / `did-stop-loading` 等） | 対象ページを埋め込み               | **削除**。同一オリジンなので埋め込み不要                                         |
| `rk.executeJavaScript(buildInjection(...))`                               | webview 内で fetch 実行            | **ページ内で直接 `fetch` に置換**（`buildInjection` の中身がほぼそのまま使える） |
| `preload.js` / `window.appApi.clearSession()`（再認証）                   | webview セッション消去             | ブラウザ標準のログアウト／再ログイン導線に置換                                   |
| 自動更新（electron-updater）                                              | GitHub Releases から更新           | **不要**（Web は配信=常に最新）                                                  |
| 本人の担当者コード自動判定（`AUTO_TANTO_PROBE`）                          | rkanri ページの DOM を走査して取得 | 自分の画面には rkanri の DOM が無い。**別手段が必要**（後述の要確認事項）        |

`<webview>` と `executeJavaScript` は **Electron 専用機能でブラウザには存在しない**。よって「現状の Electron プロジェクトをそのままブラウザで開く」ことはできないが、**上表の差し替え（主に fetch 化）を行えば、素の Web アプリとしてブラウザで動く**。しかもこの改修は WebView 迂回を外す方向なので、難度は高くない。

## ② 既存 Web サイトのサーバーに「置くだけ」で表示できるか

**結論：単なるファイルコピー（置くだけ）では動かない。ただし「同一オリジン・同一認証の配下に配置」できれば技術的には成立する。実現の可否は、サーバー側の配置権限とサイト運用者の協力に依存する。**

「置くだけで動く」ためには、少なくとも次の条件がすべて満たされる必要がある。

1. **①の改修が済んでいること**（WebView 依存を fetch に置換したブラウザ版であること）。現状ファイルをそのまま置いても `<webview>` 等が動かない。
2. **rkanri と同一オリジンの下に配置**されること（例：`https://rkanri.genech.co.jp/kanri/nippo-app/` のようなパス）。別ドメイン・別ポートだと Cookie / XSRF が効かず、①の同一オリジンの利点が失われる。
3. **既存と同じ SSO 認証の保護下に入る**こと。新しいページも Microsoft SSO（Azure AD / Entra ID）を通過してからアクセスされる配置にする。そうすればセッション Cookie と `XSRF-TOKEN` Cookie が既存画面と同じように付与され、API がそのまま通る。
4. **サーバー側のルーティング／配信設定**が必要。既存サイトは Vue の SPA（バックエンドは XSRF-TOKEN Cookie の形から Laravel と推測。`新仕様.md` §4.1 参照）で配信されているため、新しい静的ページを配信するには、その配信基盤に追加ルート・静的ファイル配置を設定する作業が発生する。
5. **CSP（Content-Security-Policy）の整合**。サーバーが独自の CSP ヘッダーを返している場合、それに合わせる必要がある（現状 `index.html` の CSP は `default-src 'self'` で同一オリジンなら問題ない見込み）。

要するに、**「同じサーバーの、同じログインの内側に、改修済みの画面を配置する」** ことができれば動く。逆に言うと、これは _ファイルを置くだけ_ ではなく **配置作業＋サーバー設定＋コード改修** の 3 点セットである。

### 最大の実務的制約：そのサーバーに手を入れられるか

技術的成立性よりも、**「そのサーバーに我々がファイルを配置・設定できるのか」** が最大の論点になる。

- rkanri が **自社管理**なら、追加ルートを切って配置すればよく、現実的。
- rkanri が **ベンダー製パッケージ／ベンダー運用**なら、勝手にファイルを追加・改変できないのが通常であり、ベンダーの許可・作業依頼が前提になる。
- → **サーバーの所有者・運用主体・改変可否の確認が最優先**（要確認事項 参照）。

## ③ 「既存サイトを本アプリの UI に置換」か「既存サイトと併用」か

選択肢を整理すると、実際には次の 3 案になる。

| 案                          | 内容                                                                                                   | 位置づけ             |
| --------------------------- | ------------------------------------------------------------------------------------------------------ | -------------------- |
| **A. 現状維持**             | Electron デスクトップアプリのまま配布                                                                  | 今のまま             |
| **B. 併用（追加配置）**     | ①②の改修版を、既存 rkanri 画面は残したまま **別ルートの Web ページとして追加**。利用者は好きな方を使う | サーバー運用の第一歩 |
| **C. 置換（全面差し替え）** | 既存 rkanri の日報画面を、本アプリの UI で **置き換える**                                              | 最終形               |

### B（併用）と C（置換）の比較

| 観点                     | B. 併用                                                    | C. 置換                                          |
| ------------------------ | ---------------------------------------------------------- | ------------------------------------------------ |
| 既存画面への影響         | なし（残す）                                               | 大（差し替え）                                   |
| リスク                   | 低い（問題があれば新ページを使わなければよい／すぐ戻せる） | 高い（不足機能・回帰・他利用者への影響）         |
| 機能網羅の要件           | 部分的でよい（できることだけ移す）                         | **既存画面の全機能を本アプリが網羅している必要** |
| 他システム／他利用者依存 | 既存画面が残るので安全                                     | 既存画面に依存する運用があると破綻               |
| ベンダー製の場合         | 追加なら比較的通しやすい                                   | 中核画面の置換は許可が下りにくい                 |
| 導入スピード             | 速い                                                       | 遅い（検証・合意形成が重い）                     |
| 保守                     | 画面が 2 つに増える（一時的な二重管理）                    | 1 つに集約できる                                 |

### 提案：まず B（併用）で始め、実績を見て C（置換）へ

**推奨は「B. 併用 → 将来的に必要なら C. 置換」の段階移行。** 理由は次のとおり。

1. **リスクが低く戻せる**：既存画面を残すので、不具合や機能不足があっても業務は止まらない。
2. **本アプリはまだ機能差分・未確認事項がある**：エラー時レスポンス仕様や本人担当者コードの取得方法など未確定点（`新仕様.md` §10）があり、いきなり既存の中核画面を置換するのは危険。
3. **併用で「同一オリジンなら WebView 無しで本当に動くか」を実地検証できる**：この検証が通れば、C（置換）も Electron 版の保守終了も現実味を持つ。
4. **将来の集約先が明確**：併用版が既存画面の全機能をカバーし、運用でも問題ないと確認できた時点で、C（置換）に進めば画面を 1 本に集約でき、Electron 配布・自動更新の保守からも解放される。

ただし C（置換）に進めるかは、**本アプリが既存画面の全機能・全利用者の使い方を網羅しているか**、**rkanri を自社で改変できるか**、の 2 点次第。ここが満たせないなら B（併用）に留めるのが妥当。

## まとめ

- **① ブラウザ表示**：画面（HTML/CSS/ロジック）はブラウザで表示・動作可能。ただし `<webview>` と `executeJavaScript` を **直接 fetch に置き換える改修**が必要。ガード済みの自動更新等はそのままでよい。改修は WebView 迂回を外す方向で、むしろ構成は単純になる。
- **② サーバー配置**：「置くだけ」では不可。**同一オリジン・同一 SSO 認証配下への配置＋サーバー側ルーティング設定＋①の改修**が揃えば動く。同一オリジンになれば Cookie / XSRF がそのまま効き、現在の WebView 迂回は不要になる。実現可否は **そのサーバーに配置・設定できるか（所有・運用主体）** に強く依存する。
- **③ 置換 vs 併用**：**まず併用（B）で追加配置して実地検証し、全機能網羅と改変権限が確認できたら置換（C）へ段階移行**するのを推奨。いきなり全面置換はリスクが高い。

## 要確認事項（次アクション）

1. **rkanri サーバーの所有・運用主体**：自社管理か、ベンダー製／ベンダー運用か。**ファイルの追加配置・ルーティング設定・（置換なら）既存画面の改変が許可されるか。** ← 最優先。
2. **配信基盤の構成**：Laravel（推測）＋ Vue SPA の配信方法。新しい静的ページ（または SPA ルート）を追加できるか。
3. **認証の配置**：新パスも既存と同じ Microsoft SSO の保護下に入るか（同じセッション Cookie / XSRF が付与されるか）。
4. **本人の担当者コードの取得方法**：現状は rkanri ページの DOM を走査（`AUTO_TANTO_PROBE`）して取得しているが、自前ページには rkanri の DOM が無い。ログインユーザーの担当者コードを返す API か、ページ埋め込み値など、**同一オリジンでの代替取得手段**が必要。
5. **CSP／セキュリティヘッダー**：サーバーが返す CSP と、本アプリの `default-src 'self'` の整合。
6. **機能網羅（置換を視野に入れる場合）**：既存 rkanri 日報画面にあって本アプリに無い機能・運用がないか。他システム／他利用者が既存画面に依存していないか。
7. **API エラー時レスポンス仕様**（`新仕様.md` §10 の未確定点）：本番運用前に確定させたい。
